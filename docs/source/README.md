# docs/source

Назначение: ссылки на исходные входные материалы, которые остаются неизменяемыми источниками для проектирования и аудита.

## Файлы

| Источник | Расположение | Статус |
| --- | --- | --- |
| Итоговое ТЗ, редакция 25.04.2026 | `itogovoe_tz_platforma_marketplace_codex.txt` | Источник истины для аудита и спорных требований |
| Стартовые правила оркестрации | `promt_start_project.txt` | Исходный orchestration prompt |
| Входное ТЗ Stage 2.1 WB API | `docs/source/stage-inputs/tz_stage_2.1.txt` | Исторический входной материал для проектирования Stage 2.1 |
| Предварительный материал Stage 2.2 Ozon API | `docs/source/stage-inputs/preliminary_stage_2_2_ozon_api.txt` | Исторический входной материал для проектирования Stage 2.2 |

Итоговое ТЗ и `promt_start_project.txt` оставлены в корне, чтобы не менять источник истины и не ломать внешние ссылки/историю передачи. Stage-input материалы лежат в `docs/source/stage-inputs/`. Исполнительные документы в `docs/` не заменяют итоговое ТЗ.

Stage-input материалы в `docs/source/stage-inputs/` не являются источником истины выше итогового ТЗ и утверждённой исполнительной документации. Они нужны для аудита происхождения решений и исторической сверки этапов.

Временные загруженные файлы, экспортированные обсуждения и черновики хранятся в `temp_docs/` до отдельного решения: перенести в `docs/source/`, оформить как task/spec/report или удалить.

## Правило чтения ТЗ

По умолчанию агент не читает итоговое ТЗ целиком. Оркестратор выдаёт конкретные разделы ТЗ только когда они нужны для задачи, аудита спорного требования или закрытия gap.
