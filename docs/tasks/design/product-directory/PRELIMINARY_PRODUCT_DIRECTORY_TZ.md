# Предварительное ТЗ: актуальный справочник товаров

Статус: предварительный черновик для обсуждения с заказчиком.

Дата: 2026-05-01.

Документ не является задачей на реализацию и не передаётся проектировщику до отдельного согласования заказчиком.

## 1. Цель

Сделать справочник товаров центральным актуальным источником данных по товарам каждого магазина и маркетплейса.

Справочник должен использоваться как базовый слой для дальнейших операций:

- цены;
- скидки;
- акции;
- остатки;
- продажи, заказы, выкупы;
- потребность в производство;
- поставки;
- отчёты;
- Excel-выгрузки.

Текущая реализация справочника не считается достаточной, потому что она работает как накопительный список товаров, которые когда-либо встретились в операциях, и может показывать неактуальное количество товаров.

## 2. Основной принцип

Один товар в справочнике должен принадлежать конкретному магазину и конкретному маркетплейсу.

Примеры:

- WB / Vital Shevron / `nmID`;
- Ozon / Vital Shevron / `product_id` или `offer_id`.

Товары разных маркетплейсов не склеиваются автоматически. Если в будущем потребуется единая внутренняя карточка физического товара, это должно проектироваться отдельным этапом.

## 3. Назначение справочника

Справочник должен:

1. Хранить актуальный каталог товаров магазина.
2. Хранить ключевые идентификаторы товара.
3. Показывать последние актуальные данные по товару.
4. Хранить историю обновлений и источники данных.
5. Связывать товар с операциями, файлами, расчётами и API-снимками.
6. Давать основу для расчётов и Excel-выгрузок.

## 4. Источники наполнения

### 4.1 WB

Для WB справочник должен наполняться и обновляться через API.

Предварительные источники:

| Данные | Источник |
| --- | --- |
| Карточки товаров | WB Content API |
| Цены и скидки | WB Prices and Discounts API |
| Остатки | WB Analytics / Stocks API |
| Заказы | WB Statistics Orders API |
| Продажи / выкупы / возвраты | WB Statistics Sales API |
| Акции | WB Promotions Calendar API, с учётом ограничений по auto promotions |

Для WB auto promotions нельзя считать API источником состава товаров акции, если API не отдаёт такой состав. Для расчёта auto promotions нужен утверждённый внешний источник состава товаров, например Excel-выгрузка из личного кабинета WB.

### 4.2 Ozon

Для Ozon справочник должен наполняться и обновляться через API.

Предварительные источники:

| Данные | Источник |
| --- | --- |
| Товары / карточки | Ozon Product API |
| Цены | Ozon Prices API |
| Остатки | Ozon Stocks API |
| Заказы / продажи | Ozon Posting / Analytics / Finance API, после уточнения доступного метода |
| Акции | Ozon Actions API |
| Эластичный бустинг | текущий Stage 2.2 Ozon Elastic Boosting API-контур |

## 5. Актуальность данных

Справочник должен быть всегда актуальным в рамках последнего успешного обновления.

Требования:

- ежедневное автоматическое обновление;
- ручной запуск обновления из веб-панели;
- фиксация времени последнего успешного обновления;
- фиксация источника обновления;
- если API недоступен, старые данные не удаляются;
- если товар не пришёл в последней успешной API-выгрузке, он не удаляется, а получает отдельный статус.

Предварительные статусы товара:

- `active` - товар найден в последнем успешном обновлении;
- `not_seen_last_sync` - товар был раньше, но не найден в последнем успешном обновлении;
- `inactive` - товар неактивен по данным источника;
- `archived` - товар архивирован вручную или по утверждённому правилу;
- `sync_error` - данные по товару не удалось обновить полностью.

Окончательные названия статусов должен предложить проектировщик, но все статусы в веб-панели должны быть человекочитаемыми.

## 6. Структура данных

Проектировщик должен спроектировать модель данных минимум для следующих сущностей.

### 6.1 Товар справочника

Минимальные поля:

- marketplace;
- store/account;
- основной внешний идентификатор;
- дополнительные внешние идентификаторы;
- артикул продавца;
- barcode;
- название;
- бренд;
- категория / предмет;
- статус;
- дата первого обнаружения;
- дата последнего успешного обновления;
- источник последнего обновления.

Для WB:

- `nmID`;
- `vendorCode`;
- barcode / `skus`;
- размерные идентификаторы, если применимо.

Для Ozon:

- `product_id`;
- `offer_id`;
- `sku`, если применимо.

### 6.2 Снимки цен

Хранить не только последнюю цену, но и историю или снимки.

Минимально:

- product;
- дата снимка;
- цена;
- цена со скидкой;
- скидка;
- валюта;
- source operation;
- source API endpoint или источник.

### 6.3 Снимки остатков

Минимально:

- product;
- дата снимка;
- общий остаток;
- остаток по складам, если источник это позволяет;
- товары в пути к клиенту / от клиента, если источник это позволяет;
- source operation.

### 6.4 Снимки продаж, заказов, выкупов

Минимально:

- product;
- период;
- количество заказов;
- количество продаж;
- количество выкупов, если источник позволяет;
- количество возвратов;
- сумма продаж, если источник позволяет;
- source operation.

Для WB учитывать ограничение: данные заказов/продаж по основным Statistics API гарантированно доступны не более чем за 90 дней.

### 6.5 Снимки участия в акциях

Минимально:

- product;
- marketplace action/promotion id;
- action name;
- участие / кандидат / не участвует;
- текущая акционная цена, если есть;
- ограничения или причина невозможности расчёта;
- source operation.

### 6.6 История изменений товара

История должна показывать:

- когда товар появился;
- когда обновился;
- какие поля изменились;
- из какой операции пришли изменения;
- из какого файла или API-снимка пришли изменения.

## 7. Связь с существующей реализацией

Текущую реализацию нельзя просто расширять как накопительный список.

Нужно:

1. Переработать текущий `MarketplaceProduct` в актуальный справочник или предложить новую модель с миграцией.
2. Сохранить совместимость с операциями, файлами, аудитом и правами.
3. Привязать существующие операции к товару справочника не только через текстовый `product_ref`, но и через ссылку на товар, если это возможно.
4. WB API prices должен обновлять актуальные товары справочника.
5. Ozon API Elastic product data должен обновлять или использовать товары справочника по утверждённым правилам.
6. Excel-операции не должны автоматически раздувать актуальный справочник, если Excel не утверждён как источник каталога.

## 8. Правило для Excel

Нужно отдельно спроектировать роль Excel в справочнике.

Предварительная рекомендация:

- API является основным источником актуального каталога;
- Excel может быть входным файлом операции;
- Excel может создавать временные строки операции;
- Excel не должен автоматически создавать актуальные товары в основном справочнике, если не выбран режим "обновить справочник из Excel";
- режим "обновить справочник из Excel" должен быть отдельным явным действием и требовать понятного подтверждения.

Этот пункт требует обсуждения с заказчиком перед проектированием.

### 9.1 Список товаров

Список товаров должен показывать:

- маркетплейс;
- магазин;
- основной артикул;
- артикул продавца;
- название;
- бренд;
- категория;
- статус;
- последняя цена;
- последний остаток;
- продажи / выкупы за выбранный период;
- дата последнего обновления.

Фильтры:

- маркетплейс;
- магазин;
- статус;
- категория;
- бренд;
- наличие остатка;
- дата последнего обновления;
- поиск по артикулу, названию, barcode.

### 9.2 Карточка товара

Карточка товара должна показывать:

- основные данные товара;
- идентификаторы;
- цены;
- остатки;
- продажи / заказы / выкупы;
- участие в акциях;
- историю изменений;
- связанные операции;
- связанные файлы;
- ошибки последней синхронизации.

### 9.3 Обновление данных

В веб-панели должны быть действия:

- обновить товары;
- обновить цены;
- обновить остатки;
- обновить продажи / выкупы за период;
- обновить акции;
- выполнить полное обновление справочника.

Во время обновления кнопки должны становиться неактивными, а пользователь должен видеть понятный статус.

### 9.4 Экспорт в Excel

Нужно сделать выгрузку данных справочника в Excel.

Пользователь должен иметь возможность:

- выбрать магазин;
- выбрать маркетплейс;
- выбрать фильтры;
- выбрать колонки;
- выгрузить Excel.

Примеры наборов колонок:

- артикулы + количество;
- артикулы + цены;
- артикулы + название + цены + остатки;
- артикулы + продажи за 90 дней + остаток;
- артикулы + потребность в производство;
- артикулы + участие в акциях.

Нужно предусмотреть сохранённые шаблоны выгрузок:

- для производства;
- для проверки цен;
- для анализа остатков;
- для ручной загрузки WB;
- для ручной загрузки Ozon.

## 10. Потребность в производство

Потребность в производство должна строиться на данных справочника, а не на разовых Excel-файлах.

Предварительная логика:

1. Обновить товары.
2. Обновить продажи / выкупы за выбранный период.
3. Обновить текущие остатки.
4. Взять настройки расчёта потребности.
5. Рассчитать среднесуточные продажи / выкупы.
6. Рассчитать целевой запас.
7. Сравнить целевой запас с остатками.
8. Сформировать потребность.
9. Выгрузить результат в Excel.

Формулы, параметры и UX расчёта потребности должны быть отдельно согласованы с заказчиком.

## 11. Автообновление

Нужно спроектировать ежедневное автоматическое обновление справочника.

Требования:

- настройка расписания;
- раздельное обновление по магазинам;
- раздельное обновление по маркетплейсам;
- фиксация успешного и неуспешного запуска;
- уведомление или отображение ошибки в панели;
- защита от параллельного запуска одного и того же обновления;
- использование последнего успешного снимка, если новый запуск завершился ошибкой.

## 12. Права доступа

Справочник должен соблюдать текущую модель object access:

- пользователь видит только товары доступных магазинов;
- владелец и администратор видят в рамках своих полномочий;
- менеджер работает с товарами доступных магазинов;
- наблюдатель может смотреть, но не менять и не запускать операции, если право не выдано.

Экспорт в Excel должен проверять права доступа.

## 13. Аудит и techlog

Нужно фиксировать:

- запуск обновления справочника;
- успешное обновление;
- ошибку обновления;
- ручную выгрузку Excel;
- изменение статуса товара;
- ручное архивирование или деактивацию;
- изменение шаблонов выгрузки;
- ошибки API;
- частичное обновление.

Секреты API не должны попадать в summary, details, audit, techlog, Excel и UI.

## 14. Приёмка

Предварительные критерии приёмки:

1. По WB Vital Shevron справочник показывает количество товаров из последнего успешного API-обновления, а не накопительный мусор из старых Excel-операций.
2. Товар, отсутствующий в последнем API-обновлении, не удаляется, но получает понятный статус.
3. Можно открыть список товаров и карточку товара.
4. Можно увидеть дату последнего обновления.
5. Можно увидеть источник данных.
6. Можно выгрузить Excel с выбранными колонками.
7. Расчёт потребности может использовать данные справочника.
8. Пользователь без доступа к магазину не видит его товары и не может выгрузить их в Excel.
9. Ошибка API не уничтожает последние успешные данные.
10. Все статусы и обозначения в веб-панели человекочитаемые.

## 15. Вопросы, требующие ответа до проектирования

Эти вопросы нельзя отдавать проектировщику на домысливание, потому что они влияют на бизнес-логику, UX веб-панели и границы будущих операций.

### Q1. Может ли Excel обновлять основной справочник товаров?

Варианты:

- A: Excel не обновляет основной справочник, а используется только как вход операции.
- B: Excel может обновить справочник только через отдельное явное действие "Обновить справочник из Excel" с подтверждением.
- C: Любая успешная Excel-операция автоматически обновляет справочник.

Рекомендация: B.

Пояснение: API должен быть основным источником актуальности, но явный импорт из Excel может понадобиться для данных, которых нет в API. Автоматический вариант C уже привёл к накоплению лишних товаров.

### Q2. Что делать с товаром, которого нет в последнем успешном API-обновлении?

Варианты:

- A: Не удалять, присвоить статус "Не найден в последнем обновлении".
- B: Автоматически архивировать.
- C: Удалять из справочника.

Рекомендация: A.

Пояснение: удаление или автоматическое архивирование опасно при сбоях API, неполных ответах или временных изменениях на стороне маркетплейса.

### Q3. Нужно ли объединять WB и Ozon товары в одну внутреннюю карточку физического товара на этом этапе?

Варианты:

- A: Нет, WB и Ozon остаются отдельными товарами разных магазинов/маркетплейсов.
- B: Да, сразу проектировать внутренний товар, который объединяет WB/Ozon/производство.

Рекомендация: A для первого этапа справочника.

Пояснение: объединение WB и Ozon потребует отдельного правила сопоставления товаров. Без него можно ошибочно связать разные товары. Но архитектуру лучше спроектировать так, чтобы позже можно было добавить внутренний товар.

### Q4. Какой источник считать главным для названия товара?

Варианты:

- A: API маркетплейса всегда главный.
- B: Excel может перезаписать название.
- C: API главный, Excel может заполнить название только если в API оно пустое.

Рекомендация: C.

Пояснение: API ближе к фактическому состоянию магазина, но Excel может быть полезен как резервный источник, если API не отдаёт название.

### Q5. Какой период по умолчанию использовать для продаж / выкупов?

Варианты:

- A: 30 дней.
- B: 60 дней.
- C: 90 дней.

Рекомендация: C для WB, потому что основные Statistics API гарантируют данные не более чем за 90 дней, а для расчёта производства обычно полезнее максимальный доступный период.

Пояснение: период должен быть изменяемым в веб-панели, но значение по умолчанию нужно зафиксировать.

### Q6. Какие показатели использовать для расчёта потребности в производство?

Варианты:

- A: Только продажи/выкупы за период и текущий остаток.
- B: Продажи/выкупы, текущий остаток, товары в пути и целевой запас в днях.
- C: Расширенная формула с сезонностью, минимальной партией, кратностью производства и ручными корректировками.

Рекомендация: B как первая рабочая версия.

Пояснение: A слишком слабый вариант, C лучше проектировать после накопления данных и проверки базовой логики.

### Q7. Нужны ли сохранённые шаблоны Excel-выгрузок?

Варианты:

- A: Да, сразу сделать шаблоны.
- B: Нет, только ручной выбор колонок.

Рекомендация: A.

Пояснение: повторяемые выгрузки для производства, цен и остатков будут использоваться часто. Шаблоны уменьшат ошибки и ускорят работу.

### Q8. Какие шаблоны Excel-выгрузок нужны первыми?

Предварительный список:

- для производства: артикул, название, продажи за период, остаток, потребность;
- для цен: артикул, название, текущая цена, скидка, цена со скидкой;
- для остатков: артикул, название, общий остаток, остатки по складам;
- для акций: артикул, название, участие в акции, акционная цена, действие.

Нужен ответ заказчика: какие из них обязательны для первой реализации и какие колонки добавить.

### Q9. Должна ли операция расчёта потребности работать без свежего обновления API?

Варианты:

- A: Разрешить расчёт на последнем успешном снимке с предупреждением.
- B: Блокировать расчёт, если данные старше установленного срока.
- C: Каждый расчёт автоматически запускает обновление API.

Рекомендация: A + настраиваемый порог устаревания данных.

Пояснение: API может быть недоступен, но бизнесу всё равно может понадобиться расчёт. При этом пользователь должен явно видеть возраст данных.

### Q10. Как показывать историю цен и остатков?

Варианты:

- A: Только таблица снимков.
- B: Таблица + график.

Рекомендация: A для первой реализации, B позже.

Пояснение: таблица быстрее в реализации и достаточна для проверки данных. Графики можно добавить после стабилизации модели данных.

### Q11. Какие операции обязаны работать только через справочник, а какие могут продолжить работу от Excel?

Предварительная рекомендация:

- цены, остатки, продажи, производство, поставки - через справочник;
- ручные Excel-сценарии скидок - оставить как отдельный путь, но связать результаты с товарами справочника;
- API-сценарии акций - переводить на справочник там, где это не ломает уже реализованную логику.

Нужен ответ заказчика перед проектированием границ миграции.

### Q12. Нужно ли показывать в интерфейсе технические источники данных?

Варианты:

- A: В рабочем интерфейсе показывать только человекочитаемые статусы, технические детали вынести в диагностику.
- B: Показывать технические поля прямо в списке/карточке товара.

Рекомендация: A.

Пояснение: это соответствует предыдущему решению по веб-панели: рабочий процесс не должен быть перегружен raw-полями.
