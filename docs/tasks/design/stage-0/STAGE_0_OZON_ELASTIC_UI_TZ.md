# Stage 0. Проектирование приведения веб-панели Ozon Elastic в порядок

Статус: audited pass, готово к передаче проектировщику  
Дата: 01.05.2026  
Инициатор: заказчик  
Оркестратор: Codex CLI  
Область: `Ozon -> Акции -> API -> Эластичный бустинг`

## 1. Назначение

Подготовить исполнительную проектную документацию для переработки веб-панели Ozon Elastic без изменения бизнес-логики Stage 2.2.

Stage 0 не является задачей на кодирование. Его результат - утверждённая проектная документация, по которой позже можно ставить задачи реализации UI.

## 3. Ограничения

- Не менять бизнес-логику расчёта Ozon Elastic.
- Не менять правила добавления, обновления и снятия товаров с акции.
- Не менять права доступа без отдельного решения.
- Не менять файловый контур и версионность файлов без отдельного решения.
- Не смешивать рабочий интерфейс, диагностику, audit и techlog.
- Не переносить технические поля в основной рабочий поток.
- Не начинать кодирование до прохождения аудита проектной документации.
- Если проектировщик обнаружит пробел по функциональности или удобству веб-панели, вопрос передаётся заказчику через оркестратора.

## 4. Утверждённые решения заказчика

### 4.1. Вкладки страницы Ozon Elastic

На странице сценария должны быть вкладки:

1. `Рабочий процесс`
2. `Результат`
3. `Диагностика`

По умолчанию открывается `Рабочий процесс`.

`Диагностика` доступна владельцу и администратору только через существующие права просмотра технических/операционных данных из `docs/product/PERMISSIONS_MATRIX.md`. Stage 0 не создаёт новых прав и не расширяет доступы. Вкладка не открывается по умолчанию и не смешивается с рабочим процессом.

### 4.2. Новая цепочка рабочего процесса

Текущие 10 шагов сокращаются до 7 операторских шагов:

1. `Скачать доступные акции`
2. `Выбрать акцию`
3. `Скачать товары и данные по ним`
4. `Обработать`
5. `Принять / не принять результат`
6. `Скачать Excel для ручной загрузки`
7. `Загрузить в Ozon`

Старые шаги 3, 4, 5 объединяются в один шаг `Скачать товары и данные по ним`:

- скачать товары, участвующие в акции;
- скачать товары-кандидаты;
- скачать данные по полученным товарам.

Старый шаг `Скачать Excel результата` убирается из основного рабочего процесса, чтобы не смешивать операторский файл ручной загрузки и технический/контрольный отчёт результата.

Это операторское объединение только на уровне UI. Оно не объединяет и не удаляет underlying Stage 2.2 operations, `Operation.step_code`, source operation links, immutable snapshots, audit records, techlog records, output links, file versions и checksums. Проектировщик обязан сохранить технический контур операций и описать маппинг 7 операторских шагов к существующим Stage 2.2 операциям.

### 4.3. Правило предупреждений объединённого шага

Если объединённый шаг `Скачать товары и данные по ним` завершился с предупреждениями, но данных достаточно для расчёта, общий шаг считается выполненным с предупреждениями и оператор может продолжать.

Блокировать продолжение можно только при ошибках, которые делают расчёт невозможным.

### 4.4. Отображение после выполнения шагов

В `Рабочем процессе` после выполнения шага показывается только короткая строка статуса:

- выполнено / выполнено с предупреждениями / ошибка;
- время;
- основные числа;
- ссылка на детали при необходимости.

Подробные summary, checksums, basis, source operations и технические коды показываются во вкладке `Диагностика`.

### 4.5. Итоги после расчёта в рабочем процессе

В `Рабочем процессе` после расчёта показываются только итоговые числа по действиям:

- `Будет обновлено в акции`
- `Будет добавлено в акцию`
- `Будет снято с акции`

Не показывать в рабочем процессе:

- товары, которые не проходят расчёт сейчас;
- кандидатов без действия;
- причины;
- ошибки строк;
- технические данные.

Эти данные относятся к вкладкам `Результат` и `Диагностика`.

### 4.6. Вкладка `Результат`

Во вкладке `Результат` строки расчёта показываются смысловыми группами:

- `Будет обновлено в акции`
- `Будет добавлено в акцию`
- `Не проходит расчёт сейчас`
- `Будет снято с акции`
- `Ошибки`

Показываются все группы. Пустые группы отображаются компактно со счётчиком `0`.

Внутри групп:

- таблица до 10 строк на страницу;
- человеко-читаемые фильтры;
- короткая человеко-читаемая причина в строке;
- полная причина раскрывается по строке.

Все обозначения данных в рабочем интерфейсе должны быть человеко-читаемыми. Технические коды остаются только в `Диагностике` и карточке операции.

### 4.7. Кандидаты

Кандидат в акцию - товар, доступный для акции, но ещё не участвующий в ней.

Если кандидат соответствует бизнес-логике и проходит расчёт, он должен попасть в группу `Будет добавлено в акцию` и добавляться в акцию при загрузке в Ozon.

Если кандидат не проходит бизнес-логику, он не показывается в `Рабочем процессе`. Его место - вкладка `Результат`, группа `Не проходит расчёт сейчас`, с причиной.

В сводке результата нужно явно показать, что цены рассчитаны по обеим группам:

- участвующие товары;
- кандидаты.

### 4.8. Принятие результата и снятие с акции

`Принять результат` остаётся одним действием review и фиксирует accepted upload basis.

Перед подтверждением нужно показать итог по группам, включая количество товаров `Будет снято с акции` и причины снятия.

Если в accepted result есть строки `Будет снято с акции` / `deactivate_from_action`, upload остаётся заблокирован до одного группового подтверждения снятия всей группы. Это подтверждение запрашивается один раз на всю группу, а не построчно. Без группового подтверждения destructive deactivate API call невозможен, add/update не должен silently proceed как финальный upload-сценарий, и write operation в Ozon не стартует.

Если товар участвует в акции, но больше не соответствует бизнес-логике, он попадает в группу `Будет снято с акции`.

### 4.9. Файлы

`Excel для ручной загрузки` показывается:

- во вкладке `Рабочий процесс`, в шаге `Скачать Excel для ручной загрузки`;
- во вкладке `Результат`, в блоке файлов результата.

`Excel результата` сохраняется, но не показывается в основном рабочем процессе. Его место:

- вкладка `Результат`;
- вкладка `Диагностика`;
- карточка операции.

Файл ручной загрузки в Stage 2.2 v1 является `manual upload Excel по Stage 1-compatible template`, а не гарантированным официальным шаблоном Ozon Elastic Boosting. Это customer-approved compatibility risk по ADR-0032. Файл должен быть явно помечен как Stage 1-compatible manual upload Excel.

Deactivate rows не должны silently omitted из manual artifacts. Если Stage 1-compatible template не может напрямую представить снятие с акции, workbook/report должен содержать отдельную вкладку или секцию `Снять с акции` с product identifiers и row-level reasons.

Отдельный Excel-отчёт результата можно делать, но не показывать в основном `Рабочем процессе`.

### 4.10. Загрузка в Ozon

Для `Загрузить в Ozon` отдельный финальный экран подтверждения не нужен.

На этой же вкладке показывается:

- итог по изменениям;
- чекбокс подтверждения;
- при наличии строк `Будет снято с акции` - отдельный group-level checkbox/confirmation для всей группы снятия;
- активная кнопка `Загрузить в Ozon` после подтверждения.

После успешной загрузки пользователь остаётся на странице сценария. Показывается финальный блок `Загрузка завершена`:

- статус загрузки;
- сколько товаров отправлено;
- сколько принято Ozon;
- сколько отклонено Ozon;
- ссылка на карточку операции;
- ссылка на Excel для ручной загрузки;
- при ошибках Ozon - группа `Отклонено Ozon` во вкладке `Результат`.

### 4.11. Карточка операции

Карточка операции `/operations/OP-.../` остаётся техническо-аудиторским экраном.

Её нужно привести в порядок:

- длинные raw JSON-like значения сворачивать;
- большие значения форматировать в scrollable/preformatted блоках;
- оставить карточку пригодной для аудита и диагностики.

### 4.12. Иерархия маркетплейсов

Страница `Маркетплейсы` должна перейти от плоского списка карточек к иерархии:

1. Маркетплейс: `WB` / `Ozon`
2. Раздел: `Цены`, `Акции`, `Остатки`, `Продажи`, `В производство`, `Поставки`
3. Режим: `Excel` / `API`
4. Сценарий, например `Эластичный бустинг`

Будущие разделы, которые ещё не реализованы, показываются как неактивные пункты с меткой `Планируется`.

Верхнюю глобальную навигацию пока не менять:

- `Главная`
- `Маркетплейсы`
- `Операции`
- `Справочники`
- `Настройки`
- `Администрирование`
- `Аудит и журналы`

## 5. Задачи Stage 0

### 0.1. Спецификация целевой страницы Ozon Elastic

Проектировщик должен подготовить структуру страницы с вкладками:

- `Рабочий процесс`
- `Результат`
- `Диагностика`

Нужно описать:

- состав каждой вкладки;
- какие данные показываются;
- какие данные скрываются;
- какие кнопки доступны;
- какие состояния есть у шага;
- какие действия блокируются при ошибках.
- какие существующие permissions управляют доступом к рабочему процессу, результату и диагностике.

Выходной документ: обновление или отдельное приложение к `docs/product/OZON_API_ELASTIC_UI_SPEC.md`.

### 0.2. Спецификация 7-шагового рабочего процесса

Проектировщик должен описать новую цепочку из 7 шагов и маппинг старых 10 шагов на новую структуру.

Особенно зафиксировать:

- объединение старых шагов 3, 4, 5;
- правило предупреждений;
- отсутствие отдельного шага `Скачать Excel результата`;
- расположение кнопки ручного Excel;
- финальный блок после загрузки.
- явное сохранение underlying Stage 2.2 operations, step codes, snapshots, audit/techlog, source operation links и file/version contour.

Выходной документ: task-scoped UI flow spec.

### 0.3. Спецификация вкладки `Результат`

Проектировщик должен описать:

- группы результата;
- счётчики;
- фильтры;
- таблицы;
- раскрытие строки;
- причины;
- отображение кандидатов;
- отображение товаров на снятие;
- файлы результата.

Выходной документ: result review spec для Ozon Elastic.

### 0.4. Спецификация вкладки `Диагностика`

Проектировщик должен описать:

- какие raw-поля и operation details переносятся из рабочего процесса в диагностику;
- как группируются технические данные;
- как отображаются checksums, basis, source operations, API metadata;
- какие ссылки ведут в карточки операций;
- какие существующие permissions дают доступ к вкладке.

Выходной документ: diagnostics spec для Ozon Elastic.

### 0.5. Спецификация карточки операции

Проектировщик должен описать изменения карточки операции:

- что остаётся в кратком блоке;
- что уходит в технические раскрываемые блоки;
- как отображать длинные значения;
- как разделить audit/debug данные и пользовательскую сводку.

Выходной документ: operation card UI amendment.

### 0.6. Спецификация иерархии `Маркетплейсы`

Проектировщик должен описать новую навигационную структуру:

- WB / Ozon;
- разделы внутри маркетплейса;
- режимы Excel/API;
- неактивные будущие пункты `Планируется`;
- текущие активные сценарии.

Выходной документ: marketplace navigation amendment.

### 0.7. Приёмочные критерии и тестовый пакет UI

Проектировщик должен подготовить acceptance checklist для будущей реализации:

- desktop и mobile;
- отсутствие горизонтального overflow на mobile;
- человеко-читаемые названия;
- отсутствие raw summary в рабочем процессе;
- сохранение файлового контура;
- сохранение прав;
- сохранение business logic;
- проверка, что пользователь не покидает страницу после действий.

Выходной документ: UI acceptance checklist для Stage 0 implementation.

### 0.8. Пакет чтения для будущей реализации

Проектировщик должен подготовить task-scoped reading package для будущих implementation-задач UI.

Пакет должен указать:

- документы, которые читает будущий UI-исполнитель;
- какие разделы Ozon Elastic specs обязательны;
- какие ADR/GAP обязательны для destructive upload/deactivate и manual Excel;
- какие документы не нужно перечитывать целиком;
- какие вопросы должны идти через оркестратора заказчику.

Выходной документ: reading package для будущей реализации Stage 0 UI.

## 6. Acceptance criteria Stage 0

Stage 0 считается готовым, если:

- подготовлена проектная документация по задачам `0.1`...`0.8`;
- аудитор подтвердил, что документация не меняет бизнес-логику Stage 2.2;
- все UX/functionality gaps либо закрыты утверждёнными решениями выше, либо вынесены заказчику;
- для будущей реализации есть task-scoped reading package;
- итоговая документация позволяет поставить отдельные implementation-задачи без домысливания.

## 7. Передача проектировщику

После аудита с результатом `pass` оркестратор передаёт этот документ проектировщику как Stage 0.

Проектировщик не пишет продуктовый код. Его задача - подготовить исполнительную документацию и список будущих implementation-задач.

## 8. Передача аудитору

Аудитор проверяет:

- соответствие правилам `AGENTS.md`;
- отсутствие кодирования до проектной задачи;
- отсутствие изменения бизнес-логики;
- полноту фиксации решений заказчика;
- отсутствие смешения рабочего UI, диагностики, audit и techlog;
- корректность маршрута unresolved UX/functionality gaps.

Ожидаемый результат аудита: `pass` или список blocking/non-blocking замечаний.
