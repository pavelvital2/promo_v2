# ARCHITECTURE.md

Трассировка: ТЗ §2-§4, §8-§10, §22, §27-§28.

## Архитектурный стиль этапа 1

Платформа проектируется как модульный монолит на утверждённом стеке ADR-0006:

- backend: Django;
- database: PostgreSQL;
- persistence layer: Django ORM поверх PostgreSQL;
- UI: server-rendered Django templates;
- frontend baseline: без отдельного SPA-фреймворка этапа 1.

Архитектурные свойства:

- одно приложение;
- одна PostgreSQL база данных;
- единая авторизация;
- единый web UI на server-rendered Django templates;
- единый execution layer;
- единая история операций;
- единый аудит;
- единый технический журнал;
- единый файловый контур.

Excel и API не являются главными архитектурными сущностями. Базовая единица модульности - бизнес-функция: "Скидки", будущие "Цены", "Остатки", "Продажи", "Поставки", "Производство", "Закупки".

## Контур deployment baseline

Обязательный production baseline этапа 1:

- Ubuntu VPS;
- PostgreSQL;
- nginx;
- systemd;
- веб-доступ через браузер;
- Django application with server-rendered web UI;
- без Docker как обязательного условия.

Отдельный staging/test контур не обязателен для ввода этапа 1, но архитектура и runbook должны позволять добавить его позже.

## Модульные границы

Минимальные доменные модули этапа 1:

| Модуль | Ответственность | Не должен делать |
| --- | --- | --- |
| Identity & Access | пользователи, роли, права, объектные доступы | считать скидки, хранить Excel-логику |
| Stores | группы/бренды, магазины/кабинеты, API-блоки | запускать операции |
| Marketplace Products | маркетплейс-товары этапа 1 | становиться производственным справочником |
| Operations & Execution | operation, run, check/process, статусы | подменять аудит и техжурнал |
| Files | метаданные, версии, физическое хранение | становиться главным пользовательским реестром |
| Discounts/WB/Excel | WB check/process и detail audit | менять Ozon или общие права |
| Discounts/Ozon/Excel | Ozon check/process и detail audit | переносить WB-параметры |
| Settings | системные и магазинные параметры | хранить результаты операций |
| Audit | значимые действия пользователей | хранить технические stack traces как основной контур |
| Tech Log | системные события, ошибки, сбои | заменять операции или аудит |
| Exports | базовые выгрузки списков и detail reports | менять исходные данные операций |

Прямые зависимости должны идти через публичные сервисы/интерфейсы модулей. Доменная логика WB/Ozon не должна зависеть от UI.

## Execution layer

На этапе 1 допускается синхронное выполнение части операций. При этом все запуски проходят через единый execution layer, который:

- создаёт operation и run-контекст;
- фиксирует статус;
- сохраняет снимок параметров;
- связывает версии файлов;
- переводит сбойные операции в статус "прервана / сбойная";
- не выполняет автоматическое возобновление после сбоя на этапе 1;
- совместим с будущими очередями и async execution.

## Расширяемость

Архитектура должна позволять добавить без перепроектирования базовых сущностей:

- API-режим скидок WB/Ozon;
- цены Excel/API;
- остатки, продажи, среднесуточные продажи;
- потребность и задания в работу;
- производство, маршруты, операторы;
- поставки, отгрузки, склады, закупки;
- автоматические сценарии и фоновые задачи.

## Безопасность

Обязательные архитектурные решения:

- вход по внутреннему логину и паролю;
- безопасное хранение password secret;
- разграничение по ролям, правам, разделам и магазинам/кабинетам;
- прямой запрет пользователя или магазина имеет приоритет над разрешением;
- роль "Владелец" не ограничивается администратором;
- API-ключи и чувствительные параметры защищены отдельными правами;
- технические детали видны только уполномоченным ролям;
- завершённые операции и аудит неизменяемы.

## Неутверждённые области

Библиотека чтения/записи Excel и детальные URL/routes Django не утверждены итоговым ТЗ. Их можно выбрать в задаче реализации при условии, что выбор:

- не меняет бизнес-логику WB/Ozon;
- не подменяет Excel-сценарий API-режимом;
- сохраняет server-rendered UI / Django templates как baseline этапа 1;
- фиксируется в документации задачи и при необходимости в ADR.
