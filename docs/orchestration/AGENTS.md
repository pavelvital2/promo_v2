# AGENTS.md

Трассировка: ТЗ §1, §2, §3, §25, §26.

## Назначение проекта

Проект - новая веб-платформа управления marketplace-операциями WB/Ozon и будущей автоматизации внутренних процессов. Обязательный этап 1: платформенный каркас и замена текущего Excel-сценария расчёта скидок WB/Ozon. Код продукта не пишется до прохождения аудита исполнительной проектной документации.

Источник истины: `itogovoe_tz_platforma_marketplace_codex.txt`, редакция от 25.04.2026. Итоговое ТЗ имеет приоритет над исполнительной документацией и используется аудитором для сверки спорных и критичных требований. Исполнители работают по утверждённой исполнительной документации и task-scoped пакетам.

## Контекст задачи

Агент перед началом задачи читает только контекст, необходимый для выполнения этой конкретной задачи:

- постановку задачи;
- корневой `AGENTS.md`;
- `docs/README.md`;
- этот документ;
- входные документы, спецификации и чек-листы, явно перечисленные оркестратором;
- role/task reading package из `docs/roles/READING_PACKAGES.md`, если он применим к задаче;
- только те разделы итогового ТЗ, которые оркестратор указал как связанные с задачей;
- связанные записи `docs/gaps/GAP_REGISTER.md` и `docs/adr/ADR_LOG.md`, если они затрагивают область задачи.

Оркестратор обязан выдавать минимальный достаточный набор входных документов и/или разделов ТЗ. Если агент видит, что выданного контекста недостаточно, он не расширяет бизнес-логику предположениями, а запрашивает уточнение у оркестратора или фиксирует gap.

## Запрет на домысливание

Нельзя самостоятельно менять или добавлять:

- бизнес-логику WB/Ozon;
- порядок правил расчёта;
- модель "Проверить / Обработать";
- модель прав, владельца и объектных ограничений;
- файловый контур и версионность файлов;
- разделение операций, аудита и технического журнала;
- reason/result codes;
- границы этапа 1;
- статус Excel как штатного и резервного режима;
- UX/functionality веб-панели, если в ТЗ или утверждённой исполнительной документации есть пробел.

Если правило отсутствует, противоречиво или неоднозначно, агент останавливает спорный участок, добавляет запись в `docs/gaps/GAP_REGISTER.md` и передаёт вопрос оркестратору.

UX/functionality gaps по веб-панели всегда идут по маршруту: агент -> оркестратор -> заказчик. Spec-blocking gaps также идут по маршруту: агент -> оркестратор -> заказчик.

Если аудитор нашёл ошибку, а проектировщик не может устранить её без пробела в ТЗ или без нового утверждённого решения, проектировщик не закрывает вопрос предположением и не оставляет его без движения. Такой вопрос фиксируется как blocking gap и передаётся заказчику через оркестратора.

## Правила изменения кода

До отдельной задачи на реализацию код продукта не создаётся. При будущей реализации агент:

- меняет только разрешённые файлы из задачи;
- не трогает несвязанные изменения;
- не удаляет чужие изменения без явного разрешения;
- сохраняет модульные границы;
- не смешивает операции, аудит и техжурнал;
- не делает API-режим вместо Excel-режима этапа 1.

## Миграции и модель данных

Миграции будущей реализации должны:

- соответствовать `docs/architecture/DATA_MODEL.md`;
- сохранять единую PostgreSQL БД этапа 1;
- фиксировать системные словари как неизменяемые пользовательским интерфейсом;
- поддерживать расширяемость под будущие модули без перепроектирования базовых сущностей.

Параллельное изменение одной миграции или одной области модели данных запрещено без координации оркестратора.

## Документация

Каждая значимая задача считается незавершённой, если не обновлена связанная документация. Изменения документации выполняются по `docs/orchestration/DOCUMENTATION_UPDATE_PROTOCOL.md`.

Исполнительная документация лежит в `docs/`; карта и правила чтения - в `docs/README.md` и `docs/DOCUMENTATION_MAP.md`. Исходное ТЗ не переносится в рабочую спецификацию и не заменяется проектными документами.

## Тестирование и аудит

Агент обязан выполнить проверки из задачи и `docs/testing/TEST_PROTOCOL.md`. Результат передаётся аудитору по `docs/audit/AUDIT_PROTOCOL.md`. Приёмка задач выполняется по `docs/testing/ACCEPTANCE_CHECKLISTS.md`.

Аудитор сверяет исполнительную документацию с исходным ТЗ. Разработчики, тестировщики и техрайтеры работают по утверждённой исполнительной документации и task-scoped пакетам.

## Отчёт агента

Итоговый отчёт должен содержать:

- что сделано;
- изменённые файлы;
- закрытые требования ТЗ;
- выполненные проверки;
- найденные проблемы;
- новые или закрытые gaps;
- следующий рекомендуемый шаг.
