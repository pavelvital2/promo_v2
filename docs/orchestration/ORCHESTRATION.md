# ORCHESTRATION.md

Трассировка: ТЗ §25, §26.

## Роль оркестратора

Оркестратор Codex CLI управляет проектом от исполнительной документации до реализации, аудита, тестирования и передачи в эксплуатацию. Оркестратор не принимает бизнес-решения вместо ТЗ и заказчика.

## Источники входа

1. `itogovoe_tz_platforma_marketplace_codex.txt` - источник истины и приоритетный документ.
2. Исполнительная проектная документация в `docs/`.
3. `docs/gaps/GAP_REGISTER.md` - активные пробелы и решения.
4. `docs/adr/ADR_LOG.md` - утверждённые архитектурные решения.
5. Состояние рабочей директории.

Итоговое ТЗ не является обязательным для полного перечитывания каждым агентом на каждую задачу. Для каждой задачи оркестратор формирует task-scoped context: минимальный достаточный набор документов, спецификаций, чек-листов, связанных gaps/ADR и конкретных разделов ТЗ, которые агент должен прочитать.

## Фаза 1: документация

Статус этой фазы: обязательная текущая фаза до начала реализации.

Порядок:

1. Проектировщик исполнительной документации готовит комплект документов.
2. Аудитор проверяет полноту, непротиворечивость, трассируемость и пригодность для Codex CLI.
3. Замечания аудитора возвращаются проектировщику.
4. Если замечание аудитора нельзя исправить без пробела в ТЗ или без утверждённого решения заказчика, проектировщик фиксирует blocking gap и передаёт вопрос оркестратору для эскалации заказчику.
5. Цикл повторяется до статуса: "документация готова к реализации".
6. Оркестратор формирует задачи реализации этапа 1.

Комплект задач реализации хранится в `docs/tasks/implementation/stage-1/IMPLEMENTATION_TASKS.md` и `docs/tasks/implementation/stage-1/`. Эти файлы являются практическим task-scoped входом для будущих агентов реализации после повторного аудита документации и соблюдения phase gates из `docs/gaps/GAP_REGISTER.md`.

## Фаза 2: реализация этапа 1

Реализация допускается только после аудита документации. Обязательный объём:

- каркас платформы;
- веб-панель;
- пользователи, роли, права, доступы, объектные ограничения;
- магазины / кабинеты / подключения;
- операции, run, проверки и обработки;
- файловый контур;
- аудит и технический журнал;
- модуль "Скидки" Excel для WB и Ozon;
- настройки WB;
- базовые экспорты;
- эксплуатационный минимум.

Не входят в прикладную реализацию этапа 1: API-скидки, загрузка в кабинеты через API, цены, остатки, продажи, производство, поставки, закупки.

## Маршрутизация задач

| Тип задачи | Агент | Обязательный вход | Обязательный выход |
| --- | --- | --- | --- |
| Проектирование документации | Проектировщик исполнительной документации | Задача, task-scoped разделы ТЗ, профильная документация | Документы, gaps, ADR |
| Уточнение пробела | Внутренний проектировщик | GAP-запись | Уточнение или вопрос заказчику |
| Реализация | Разработчик | Задача, спецификации, чек-лист | Код, миграции, тесты, отчёт |
| Аудит | Аудитор | Изменения, task-scoped разделы ТЗ, документация | Pass/fail, замечания, gaps |
| Тестирование | Тестировщик | Спецификация, тест-протокол | Результаты прогонов, дефекты |
| Документирование | Техрайтер | Изменения и отчёты | Обновлённые документы |

## Правило gaps

Если агент обнаружил пробел или противоречие:

1. Не реализует спорный участок.
2. Создаёт запись в `docs/gaps/GAP_REGISTER.md`.
3. Передаёт оркестратору.
4. Оркестратор назначает внутреннего проектировщика или передаёт вопрос заказчику.
5. Работа возобновляется только после фиксации решения в `docs/gaps/GAP_REGISTER.md` и при необходимости `docs/adr/ADR_LOG.md`.

Если gap возник из аудиторского замечания и проектировщик не может устранить его без нового решения, оркестратор не возвращает задачу в бесконечный цикл исправлений. Оркестратор переводит вопрос в маршрут customer escalation: `проектировщик -> оркестратор -> заказчик`, фиксирует статус в `docs/gaps/GAP_REGISTER.md` и блокирует затронутый участок до решения.

Если gap связан с функциональностью или удобством использования веб-панели и не закрыт ТЗ/утверждённой исполнительной документацией, оркестратор не поручает агенту принимать UX/functionality решение самостоятельно. Такой вопрос передаётся заказчику через оркестратора и фиксируется в `docs/gaps/GAP_REGISTER.md`.

## Контроль завершения задачи

Задача не завершена, если отсутствуют:

- ссылка на требования ТЗ;
- перечень фактически использованных входных документов и разделов ТЗ;
- перечисление изменённых файлов;
- результаты проверок;
- статус документации;
- статус gaps;
- передача следующему агенту.
