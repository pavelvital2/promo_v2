# DOCUMENTATION_UPDATE_PROTOCOL.md

Трассировка: ТЗ §25-§26.

## Когда обновлять документацию

Документация обновляется при каждом значимом изменении:

- модели данных;
- прав;
- операций;
- файлового контура;
- WB/Ozon логики;
- UI-сценариев;
- тестов и приёмки;
- эксплуатации;
- архитектурных решений;
- закрытия или появления gaps.

## Правила

- Итоговое ТЗ не редактируется агентами.
- Итоговое ТЗ остаётся источником истины, но документационные задачи выполняются по task-scoped контексту: оркестратор указывает нужные документы и конкретные разделы ТЗ.
- Изменение бизнес-правил возможно только после утверждённого решения.
- Новое решение фиксируется в `docs/adr/ADR_LOG.md`.
- Пробел фиксируется в `docs/gaps/GAP_REGISTER.md`.
- Профильная спецификация обновляется в том же изменении, что и код.
- Если исправление аудиторского замечания требует решения, отсутствующего в ТЗ и утверждённой документации, проектировщик фиксирует blocking gap и передаёт вопрос заказчику через оркестратора.

## Минимальный отчёт техрайтера

```md
Что обновлено:
Почему обновлено:
Связанные требования ТЗ:
Использованные входные документы / разделы ТЗ:
Изменённые документы:
Связанные ADR/GAP:
Оставшиеся ограничения:
```

## Запреты

Нельзя:

- ссылаться на устные договорённости;
- заменять формальные правила фразами "как в старой программе";
- скрывать спорное решение внутри текста без GAP/ADR;
- оставлять spec-blocking аудиторское замечание без эскалации через оркестратора;
- описывать будущий этап как обязательную реализацию этапа 1.
