# ACCEPTANCE_CHECKLISTS.md

Трассировка: ТЗ §24-§27.

## Чек-лист документации фазы 1

- Итоговое ТЗ указано как источник истины.
- Обязательный этап 1 отделён от будущих этапов.
- Описаны архитектура, модель данных, UI, права, операции, файлы, WB/Ozon Excel, аудит, тесты, эксплуатация.
- Есть агентный комплект документов.
- Все неоднозначности вынесены в `docs/gaps/GAP_REGISTER.md`.
- Нет ссылок на скрытый контекст, переписку или старую программу как источник правил.
- Есть трассировка на разделы ТЗ.

## Чек-лист реализации задачи

- Задача имеет ID, границы, входные документы и требования ТЗ.
- Изменены только разрешённые файлы.
- Не нарушены запреты ТЗ §1.4 и §26.8.
- Обновлена связанная документация.
- Добавлены или обновлены тесты по риску изменения.
- Отчёт содержит проверки, изменённые файлы и gaps.

## Чек-лист WB Excel

- Принимается ровно 1 файл цен и 1-20 файлов акций.
- Поддерживается только `.xlsx`.
- Проверены лимиты 25 МБ на файл, 100 МБ на run, максимум 20 promo files.
- Check ничего не пишет в workbook и не создаёт output.
- Process меняет только `Новая скидка`.
- Используется точная десятичная арифметика и `ceil`.
- Сохранён снимок WB-параметров и их источников.
- Check/process используют одинаковую нормализацию.
- Системные defaults WB: `wb_threshold_percent = 70`, `wb_fallback_over_threshold_percent = 55`, `wb_fallback_no_promo_percent = 55`.
- Итоговая скидка вне 0..100 даёт row error `wb_discount_out_of_range`, check с ошибками и запрет process без обрезки/частичной обработки.
- Detail rows используют только утверждённый закрытый каталог WB reason/result codes.

## Чек-лист Ozon Excel

- Принимается ровно 1 `.xlsx`.
- Есть лист `Товары и цены`.
- Данные читаются со строки 4.
- Проверены колонки J, K, L, O, P, R.
- Check ничего не пишет в workbook и не создаёт output.
- Process меняет только K и L.
- В K пишется только `Да` или пусто.
- В L пишется число или пусто.
- Все ветки decision rules покрыты тестами.

## Чек-лист операций

- Проверка и обработка являются отдельными операциями.
- Обработка хранит ссылку на проверку-основание.
- Актуальность проверки сверяется по marketplace, store, files, file versions, parameters, logic version.
- Завершённые операции неизменяемы.
- Повторная проверка/обработка создаёт новую операцию.
- Предупреждения делятся на информационные и подтверждаемые.

## Чек-лист acceptance artifacts

- Реальные контрольные WB/Ozon файлы переданы заказчиком.
- Checksums исходных файлов зафиксированы.
- Результаты старой программы зафиксированы.
- Expected summary и expected row-level results заполнены.
- Edge-case наборы подготовлены отдельно от реальных файлов.
- Нет фиктивных customer files/checksums/expected results.

## Чек-лист эксплуатации

- PostgreSQL backup выполняется ежедневно.
- Backup серверного файлового хранилища выполняется ежедневно.
- Backup retention составляет 14 дней.
- Перед production update выполнен обязательный backup PostgreSQL и файлового хранилища.
- Restore check выполнен по документированной ручной процедуре после setup и перед важным релизом.
- Audit records и techlog records имеют retention 90 дней.
- Очистка audit/techlog выполняется только регламентной процедурой, не через обычный UI.
