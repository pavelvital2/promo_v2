# PROJECT_DOCUMENTATION_PLAN.md

Трассировка: ТЗ §25, §28.

## Цель комплекта

Комплект исполнительной проектной документации должен быть самодостаточным для реализации этапа 1 агентами Codex CLI. Документация не заменяет итоговое ТЗ и не имеет права изменять его бизнес-правила.

Итоговое ТЗ остаётся источником истины для проектирования и аудита, но агентные задачи должны выполняться по task-scoped входному контексту. Оркестратор указывает минимальный набор документов и конкретных разделов ТЗ, достаточный для задачи, чтобы агент не перечитывал всё ТЗ при каждом поручении.

## Границы этапа 1

Обязательно проектируется для реализации:

- модульный монолит, единая PostgreSQL БД, единый UI/API;
- пользователи, роли, права, объектные ограничения;
- магазины / кабинеты / подключения;
- операции, run, проверки, обработки;
- файловый контур с версиями и сроками хранения;
- аудит и технический журнал;
- WB/Ozon скидки через Excel;
- настройки расчёта WB;
- карточки операций, товаров, магазинов;
- базовые экспорты;
- эксплуатационный минимум: backup, restore, update, system notifications.

Будущие этапы фиксируются только как архитектурная совместимость: API-режимы, цены, остатки, продажи, производство, поставки, закупки.

## Критерии готовности документации

- Каждый обязательный раздел ТЗ этапа 1 отражён хотя бы в одном профильном документе.
- Для WB/Ozon приведены правила check/process без домысливания.
- Для будущих этапов указана только архитектурная подготовка.
- Открытые вопросы не спрятаны в тексте, а вынесены в `docs/gaps/GAP_REGISTER.md`.
- Документы содержат входы/выходы для разработчика, аудитора и тестировщика.
- Агентные инструкции и шаблоны явно поддерживают task-scoped чтение контекста, эскалацию spec-blocking вопросов и UX/functionality gaps веб-панели заказчику через оркестратора.
- Комплект задач реализации содержит явный индекс, порядок работ, зависимости, разрешённые области изменений, required checks и handoff format.
