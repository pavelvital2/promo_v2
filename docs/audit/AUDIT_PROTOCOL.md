# AUDIT_PROTOCOL.md

Трассировка: ТЗ §20, §24-§27.

## Цель аудита

Аудит проверяет соответствие итогов задачи итоговому ТЗ, исполнительной документации, границам этапа 1 и запрету на домысливание.

Аудитор сверяет результат с task-scoped входным контекстом: изменениями, профильной документацией и разделами ТЗ, указанными оркестратором для задачи. Если для вывода нужен дополнительный раздел ТЗ или документ, аудитор запрашивает его через оркестратора, а не требует от каждого агента полного перечитывания итогового ТЗ.

## Обязательные виды аудита

- документационный аудит;
- архитектурный аудит;
- аудит модели данных и миграций;
- аудит прав и безопасности;
- аудит операций/run/files;
- аудит WB/Ozon бизнес-логики;
- аудит UI-соответствия;
- аудит тестов;
- аудит эксплуатационного контура.

## Проверки аудитора

Аудитор проверяет:

- нет ли неутверждённых бизнес-правил;
- не смешаны ли операции, аудит и техжурнал;
- соответствуют ли audit action codes и techlog event types `docs/architecture/AUDIT_AND_TECHLOG_SPEC.md`;
- соблюдается ли retention audit/techlog 90 дней и запрет обычного UI-удаления;
- check/process разделены;
- завершённые операции неизменяемы;
- объектные ограничения по магазинам соблюдены;
- владелец защищён от ограничения;
- WB/Ozon Excel логика соответствует профильным спецификациям;
- Ozon не получил WB-параметры;
- Excel не заменён API;
- документация обновлена;
- gaps заведены для спорных областей.
- spec-blocking вопросы после аудита переданы оркестратору для решения или эскалации заказчику.

## Критерии fail

Fail обязателен, если найдено:

- изменение WB/Ozon бизнес-логики без утверждения;
- объединение check/process;
- отсутствие object access;
- возможность ограничить владельца администратором;
- потеря file version links;
- изменение запрещённых Excel-колонок;
- отсутствие тестов для затронутой критичной логики;
- реализация будущего API как замена Excel этапа 1;
- попытка закрыть аудиторское замечание предположением вместо GAP/ADR/эскалации заказчику.
