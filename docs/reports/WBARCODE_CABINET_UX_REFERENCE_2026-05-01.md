Дата: 2026-05-01.

Статус: обзор внешнего сервиса как UX/reference material для будущего справочника товаров, печати этикеток и веб-панели Promo v2.

## 2. Главные UX-находки

### 2.1 Постоянная левая навигация

В кабинете используется постоянная левая навигация:

- Главная;
- Наклейки;
- Магазины;
- Wildberries;
- Ozon;
- Яндекс Маркет;
- Честный знак;
- Модули;
- Полезное;
- Мой аккаунт.

При входе в маркетплейс раскрываются его рабочие подразделы.

Пример WB:

- Товары;
- FBS - Сборочные задания;
- FBS - Поставки Маркетплейс;
- FBW - Поставки;
- Коробки/Палеты/Поставки;
- Управление ценами;
- Переименование товаров;
- Генерация баркодов;
- Платное хранение;
- Цены конкурентов.

Пример Ozon:

- Товары;
- FBS - отгрузки;
- FBS - поставки из файла;
- FBO - поставки;
- FBO - поставки из файла.

Что перенять:

- Для Promo v2 нужна такая же иерархия: marketplace -> рабочие модули.
- Для каждого маркетплейса список подразделов должен быть устойчивым и одинаково доступным из любой страницы.
- Активный раздел должен быть явно подсвечен.

### 2.2 Магазины как входная точка к данным

Раздел "Магазины" устроен как список карточек магазинов.

В каждой карточке видно:

- маркетплейс;
- имя магазина;
- дата создания;
- дата последнего обновления;
- количество товаров;
- проверка подключения;
- обновление товаров;
- ссылки/иконки на редактирование и удаление;
- YML, если применимо.

Что перенять:

- В Promo v2 карточка магазина должна показывать состояние API-подключения и свежесть данных.
- Количество товаров должно идти из актуального справочника, а не из накопительного списка.
- Должна быть отдельная кнопка "Обновить товары".
- Для каждого магазина нужно показывать дату последнего успешного обновления.

### 2.3 Товарный список как рабочий экран

WB товарный список показывает:

- общее количество товаров;
- поиск по названию / артикулу / бренду;
- сортировку;
- фильтры;
- переключатель "использовать ранее созданные наклейки";
- таблицу товаров;
- картинку товара;
- название;
- категорию;
- артикул продавца;
- цвет/размер/габариты;
- цену и скидку;
- остатки;
- конверсии;
- комиссию;
- barcode;
- артикул WB;
- артикул wbarcode;
- магазин;
- количество для печати;
- действия для создания наклеек / файла поставки / очистки выбора.

Ozon товарный список похож, но адаптирован под Ozon:

- Ozon Id;
- SKU;
- цена;
- скидка;
- цена без скидки;
- остатки;
- barcode/label code;
- магазин;
- состав;
- количество.

Что перенять:

- Справочник товаров в Promo v2 должен быть не техническим списком, а рабочей таблицей.
- В таблице нужна картинка товара, если API её отдаёт.
- Колонка "Количество" должна быть прямо в списке, если пользователь готовит этикетки/поставку/производство.
- Для WB и Ozon нужны разные наборы идентификаторов, но единый принцип отображения.
- Нужна пагинация и выбор размера страницы.

### 2.4 Быстрый переход от справочника к печати

В списке товаров можно сразу выбрать количество и перейти к созданию наклеек.

Что перенять:

- Справочник Promo v2 должен быть базой для операций, а не отдельной справочной страницей.
- Из списка товаров должны запускаться действия:
  - выгрузить Excel;
  - создать этикетки;
  - сформировать потребность;
  - подготовить поставку;
  - перейти к ценам/акциям по выбранным товарам.

### 2.5 Раздел создания наклеек

Раздел "Создать наклейки" устроен как рабочий редактор:

- таблица ввода barcode / артикул / название / количество;
- кнопки "добавить ещё наклейку", "установить количество для всех", "очистить", "обнулить количество";
- настройки печати на той же странице;
- выбор шаблона этикетки с визуальным preview;
- выбор формата штрихкода CODE-128 / EAN-13;
- выбор бумаги A4 / термоэтикетка;
- размер этикетки;
- размер текста;
- межстрочный интервал;
- версия генератора;
- чекбоксы дополнительных опций;
- понятная ошибка, если barcode и количество не заполнены;
- кнопки скачать / открыть в новой вкладке / сохранить.

Что перенять:

- Будущий модуль этикеток в Promo v2 должен работать на одной странице: таблица товаров + настройки печати + результат.
- Настройки печати должны быть визуальными, с preview шаблона.
- Кнопки результата должны быть рядом с рабочим процессом, а не на отдельной странице.
- Валидация должна быть человекочитаемой.

### 2.6 Поиск по файлу

Раздел позволяет загрузить файл с артикулами или штрихкодами и найти этикетки.

Логика:

1. выбрать магазин;
2. выбрать поле поиска: артикул или штрихкод;
3. загрузить файл;
4. получить найденные этикетки.

Что перенять:

- Для Promo v2 полезен сценарий "найти товары из файла в справочнике".
- Это применимо не только к этикеткам, но и к ценам, акциям, производству и поставкам.

### 2.7 Подсказки внутри рабочих страниц

В рабочих разделах есть короткие пояснения и FAQ:

- какой файл можно загрузить;
- где скачать файл в личном кабинете маркетплейса;
- какие ограничения у API;
- почему перед поставками нужно обновить товары.

Что перенять:

- Подсказки нужны рядом с действием.
- Не нужно перегружать основной workflow raw-данными.
- Технические детали и диагностику лучше держать отдельно.

## 3. Что особенно важно для Promo v2

### 3.1 Справочник должен быть центром системы

Wbarcode явно построен вокруг импортированных товаров:

- магазины обновляют товары;
- товары используются для наклеек;
- поставки требуют обновлённые товары;
- поиск по файлу ищет товары среди импортированных;
- цены/остатки/аналитика показываются в товарном списке.

Для Promo v2 это подтверждает выбранное направление:

```text
API/Excel sync -> справочник товаров -> рабочие операции
```

### 3.2 Свежесть данных должна быть видна

В wbarcode на карточке магазина видно "Обновлен: дата/время".

Для Promo v2 нужно показывать:

- когда обновлены товары;
- когда обновлены цены;
- когда обновлены остатки;
- когда обновлены продажи;
- какой источник последнего успешного обновления;
- есть ли ошибка последней синхронизации.

### 3.3 Количество товаров должно быть контекстным

Wbarcode показывает разные количества по магазинам и маркетплейсам.

Для Promo v2 нельзя показывать накопительное количество всех когда-либо встреченных строк. Нужно показывать:

- актуальные товары;
- не найденные в последнем обновлении;
- архивные;
- с ошибками синхронизации.

### 3.4 Печать этикеток нужно проектировать как отдельный будущий модуль

Функциональность wbarcode по этикеткам полезна для Promo v2, но это отдельный крупный модуль.

Минимальный будущий scope:

- создание этикеток из справочника;
- создание этикеток из Excel;
- выбор шаблона;
- выбор размера;
- CODE-128 / EAN-13;
- термоэтикетка / A4;
- печать количества;
- сохранение набора этикеток;
- архив этикеток;
- поиск этикеток по файлу;
- создание файла поставки, если это нужно.

## 4. Что не стоит копировать буквально

### 4.1 Перегрузка боковой панели

Левая навигация удобная, но внизу есть рекламные/промо-блоки, которые занимают много места.

Для Promo v2 лучше:

- оставить рабочую навигацию;
- не добавлять рекламные карточки;
- использовать нижнюю часть сайдбара только для статуса системы или кратких уведомлений.

### 4.2 Cookie popup перекрывает рабочие кнопки

На скриншотах cookie popup перекрывает нижнюю панель действий.

Для Promo v2:

- системные уведомления не должны закрывать рабочие кнопки;
- нижняя панель действий должна оставаться доступной.

### 4.3 Некоторые значения не человекочитаемы

В Ozon списке встречается `NaN%` и `не число ₽`.

Для Promo v2 такие значения должны отображаться как:

- "нет данных";
- "не рассчитано";
- "ошибка цены";
- "не получено из API".

Сырые `NaN`, `null`, `None`, `undefined` в рабочем интерфейсе запрещены.

### 5.2 Карточка магазина

Показывать:

- marketplace;
- имя магазина;
- тип подключения;
- статус подключения;
- дата последнего обновления товаров;
- товары: актуальные / не найдены / ошибки;
- цены: дата последнего обновления;
- остатки: дата последнего обновления;
- продажи: дата последнего обновления;
- действия: проверить подключение, обновить товары, открыть товары, настройки.

### 5.3 Список товаров

Сделать плотную рабочую таблицу:

- изображение;
- название;
- артикул продавца;
- marketplace id;
- barcode;
- магазин;
- цена;
- скидка;
- остаток;
- продажи/выкупы за период;
- статус;
- дата обновления;
- количество для выбранной операции.

Действия:

- экспорт;
- создать этикетки;
- сформировать потребность;
- перейти к ценам;
- перейти к акциям;
- открыть карточку товара.

### 5.4 Карточка товара

Показывать:

- основные идентификаторы;
- изображение;
- цены;
- остатки;
- продажи/выкупы;
- участие в акциях;
- история обновлений;
- связанные операции;
- связанные файлы;
- ошибки синхронизации.

### 5.5 Экспорт

Добавить экспорт из справочника с выбором колонок и шаблонами:

- производство;
- цены;
- остатки;
- акции;
- этикетки;
- поставки.

### 5.6 Этикетки

Добавить в будущий backlog модуль "Этикетки":

- создание из справочника;
- создание из файла;
- сохранённые наборы;
- архив;
- шаблоны печати;
- preview;
- настройка размера/формата;
- проверка обязательных полей.

## 6. Вывод

Wbarcode подтверждает, что для Promo v2 справочник товаров должен быть не второстепенной страницей, а центральной рабочей точкой системы.

Главное, что стоит перенять:

1. Магазины как источник актуальных товаров.
2. Видимая дата обновления.
3. Товарная таблица с изображением, ценами, остатками и идентификаторами.
4. Быстрый переход от товаров к операциям.
5. Печать этикеток как workflow на одной странице.
6. Поиск товаров по файлу.
7. Настройки результата рядом с рабочими данными.

Для Promo v2 это нужно учитывать при проектировании актуального справочника товаров и будущего модуля этикеток.
