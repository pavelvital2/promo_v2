## Ключевой вывод

Панель уже рабочая, но сейчас она выглядит как технический протокол, а не как операторский интерфейс. Главная проблема не в цветах или шрифтах, а в информационной архитектуре: действия, статусы, служебные snapshot-поля, технические коды, файлы и диагностические таблицы выводятся рядом и конкурируют за внимание.

Самый проблемный участок: `Ozon -> Акции -> API -> Эластичный бустинг`. Эта страница одновременно является мастером процесса, карточкой диагностики, списком операций, витриной файлов, таблицей строк и экраном подтверждения. Из-за этого оператору трудно понять, что уже сделано, что нужно нажать сейчас и какой результат можно скачать.

## Findings

### P1. Ozon Elastic перегружен технической информацией

На странице `Ozon API Elastic` выведены:

- 10 шагов процесса;
- блок подключения;
- блок текущего состояния;
- счётчики;
- summary по каждому шагу;
- технические поля `basis`, `downstream_basis_source`, `selection_basis`, `source_operations`, `diagnostics_counts`;
- Review rows;
- Deactivate confirmation.

В результате главный сценарий теряется. Пользователь видит длинную страницу с большим количеством технических значений, хотя по сути ему нужен пошаговый процесс:

1. Скачать доступные акции.
2. Выбрать акцию.
3. Скачать участвующие товары.
4. Скачать кандидатов.
5. Скачать данные по товарам.
6. Обработать.
7. Принять или не принять результат.
8. Скачать Excel результата / Excel для ручной загрузки.
9. Загрузить в Ozon.

Рекомендация: сделать Ozon Elastic отдельным мастером, где каждый шаг показывает только человеко-читаемый статус, дату, результат и основную кнопку. Технические details убрать в раскрываемый блок "Диагностика".

### P1. Мобильная версия Ozon Elastic сломана горизонтальным overflow

Playwright mobile viewport `390x844` показал:

- `clientWidth`: 390;
- `scrollWidth`: 1065;
- `overflow`: true.

Главная причина: таблицы и длинные JSON-like значения не адаптируются. На мобильной ширине Review rows и summary-таблицы уходят вправо, часть интерфейса визуально недоступна без горизонтальной прокрутки.

Рекомендация: для мобильной ширины:

- таблицы переводить в карточки строк или включать локальный горизонтальный scroll внутри таблицы, а не всей страницы;
- длинные технические значения скрывать в `<details>`;
- Review rows на мобильном показывать как список из 10 карточек с ключевыми полями.

### P1. Карточка операции выводит слишком много raw-данных

`/operations/OP-2026-000040/` содержит 55 687 символов текста на странице. Основной источник перегруза - `Summary` и `Детализация строк`, где в поле `Value` выводятся крупные словари/структуры данных.

Карточка операции должна быть аудиторской и диагностической, но сейчас она плохо читается даже как диагностика: длинные значения ломают таблицу и вытягивают страницу на десятки экранов.

Рекомендация:

- в карточке операции оставить краткое summary: статус, шаг, магазин, файлы, ошибки/warnings, result_code;
- raw summary показывать только в блоке "Технические данные" с раскрытием;
- большие JSON-like значения форматировать моноширинным блоком с ограничением высоты и кнопкой "Показать полностью";
- детализацию строк для API-операций разделить на "Ошибки и предупреждения" и "Сырые строки".

### P1. В панели смешаны пользовательские и технические термины

На рабочих страницах видны термины:

- `mode=api / marketplace=ozon / module=ozon_api`;
- `result_code`;
- `source_group`;
- `upload_ready`;
- `planned_action`;
- `deactivate_required`;
- `not_applicable`;
- `completed_success`;
- `completed_with_warnings`;
- `basis`, `checksum`, `source_operation_id`.

Для аудитора и разработчика это нормально. Для владельца/администратора магазина это шум.

Рекомендация: в операторском интерфейсе показывать русские бизнес-формулировки, а технические коды держать вторым уровнем:

- `completed_success` -> "Успешно";
- `completed_with_warnings` -> "Успешно, есть предупреждения";
- `interrupted_failed` -> "Не выполнено";
- `upload_ready=True` -> "Готово к загрузке";
- `update_action_price` -> "Обновить акционную цену";
- `skip_candidate` -> "Не добавлять кандидата";
- `deactivate_from_action` -> "Снять с акции".

### P2. Нет явной иерархии маркетплейс -> раздел -> режим

Страница `Маркетплейсы` уже содержит 4 карточки, но это плоский список:

- WB -> Скидки -> Excel;
- WB -> Скидки -> API;
- Ozon -> Скидки -> Excel;
- Ozon -> Акции -> API -> Эластичный бустинг.

Это не масштабируется под будущие разделы: цены, акции, остатки, продажи, производство, поставки.

Рекомендация: заложить навигационную модель:

- уровень 1: маркетплейс `WB` / `Ozon`;
- уровень 2: область `Цены`, `Акции`, `Остатки`, `Продажи`, `В производство`, `Поставки`;
- уровень 3: режим `Excel` / `API`;
- уровень 4: конкретный сценарий, например `Эластичный бустинг`.

На текущем этапе можно не делать весь будущий функционал, но каркас навигации уже стоит поменять, чтобы Stage 2.2 не закрепил плоскую структуру.

### P2. Главная страница дублирует навигацию и не отвечает на вопрос "что делать сейчас"

Главная показывает:

- доступные разделы;
- быстрые действия;
- системные уведомления;
- последние операции;
- требуют внимания;
- сроки хранения файлов.

Это полезно, но нет приоритета. Блок "Требуют внимания" визуально похож на обычные операции, а быстрые действия дублируют разделы.

Рекомендация:

- сверху показывать рабочие сценарии по магазинам: `Vital Shevron / Ozon / Эластичный бустинг / последнее состояние`;
- отдельно выделить операции, где требуется решение пользователя;
- сроки хранения файлов перенести ниже или в отдельный раздел файлов/операций.

### P2. Шаги процесса выглядят как независимые карточки, а не как единый процесс

В Ozon Elastic каждый шаг оформлен одинаковой большой панелью. Это создаёт ощущение, что все шаги одинаково важны и доступны одновременно.

Рекомендация:

- сделать вертикальный stepper;
- для завершённых шагов показывать компактную строку: номер, название, статус, время, файл/результат;
- текущий доступный шаг раскрывать;
- будущие шаги делать свернутыми и неактивными;
- действие на шаге держать справа или внизу в одинаковом месте.

### P2. Кнопки имеют базовое состояние, но не хватает уверенности после нажатия

В CSS уже есть `data-loading-text` и стили `.is-submitting`, но визуально на скриншотах кнопки выглядят одинаково как обычные формы. Для длинных API-операций оператору важно видеть, что действие принято и выполняется.

Рекомендация:

- после submit блокировать все кнопки текущего шага;
- менять текст кнопки на "Выполняется...";
- показывать inline spinner или статусную строку "Операция запущена";
- после завершения возвращать пользователя на ту же страницу и раскрывать следующий шаг.

### P2. Счётчики требуют перевода в бизнес-сводку

Текущий блок `Счётчики` полезен, но показывает технические ключи:

- `actions downloaded`;
- `elastic actions found`;
- `active products count`;
- `candidate products count`;
- `stock rows count`;
- `skip candidate count`.

Рекомендация: сделать блок "Итог по акции":

- "Доступных акций: 5";
- "Эластичный бустинг: 1";
- "Участвуют в акции: 484";
- "Кандидаты: 61";
- "Будет обновлено: 484";
- "Будет добавлено: 0";
- "Будет снято с акции: 0";
- "Не проходят расчёт сейчас: 61";
- "Предупреждения: 61".

Технические названия оставить в диагностике.

### P3. Визуальная система слишком базовая, но её не нужно менять первой

Цвета, типографика и карточки сейчас простые, но читаемые. Сначала нужно исправить структуру и плотность. Полный визуальный редизайн без изменения информационной архитектуры не решит главную проблему.

Рекомендация: первый UI-этап должен быть не "красивее", а "понятнее":

- меньше raw-полей;
- больше статусов и итогов;
- один текущий следующий шаг;
- файлы рядом с результатом;
- диагностика свернута.

## Предлагаемый порядок работ

### Этап UI-1. Привести в порядок Ozon Elastic

Цель: сделать рабочий мастер для Stage 2.2.

Состав:

- разделить экран на `Рабочий процесс`, `Итог`, `Файлы`, `Проверка строк`, `Диагностика`;
- сделать stepper из 10 шагов;
- скрыть raw summary в diagnostics;
- оставить Review rows на 10 строк;
- добавить человеко-читаемые labels;
- исправить mobile overflow;
- сохранить все текущие операции и права без изменения бизнес-логики.

### Этап UI-2. Привести карточку операции к двум режимам

Цель: операция должна быть понятна владельцу, но полезна аудитору.

Состав:

- режим "Кратко";
- режим "Технические данные";
- raw values в `<details>` / scrollable `<pre>`;
- отдельный блок файлов;
- отдельный блок ошибок и warnings.

### Этап UI-3. Перестроить навигацию маркетплейсов

Цель: заложить иерархию:

`Маркетплейс -> область -> режим -> сценарий`.

На первом шаге достаточно перестроить `Маркетплейсы` и верхнюю навигацию, не создавая пустые будущие модули.

### Этап UI-4. Унифицировать WB API, WB Excel, Ozon Excel

Цель: привести все рабочие страницы к одной модели:

- выбор магазина;
- текущий черновик/последний результат;
- основные действия;
- файлы;
- диагностика.

## Вопросы, которые нельзя решать без заказчика

Эти вопросы относятся к UX/functionality веб-панели и по правилам проекта должны идти заказчику:

1. Нужен ли владельцу магазина доступ к технической диагностике на рабочих страницах, или её показывать только администратору/аудитору?
2. Должна ли страница Ozon Elastic показывать все 10 шагов всегда, или только текущий/следующий шаг с историей ниже?
3. Нужно ли делать отдельный раздел "Файлы" внутри каждого сценария, или достаточно файлов рядом с результатом шага?
4. Какой режим по умолчанию в карточке операции: краткий пользовательский или полный технический?

## Рекомендация

Начать с `Этап UI-1: Ozon Elastic`. Это самый перегруженный и самый важный экран Stage 2.2. После него ту же модель можно перенести на WB API и Excel-страницы.

Технически это лучше делать как отдельную проектную задачу: сначала проектировщик фиксирует целевую структуру страницы и список скрываемых/показываемых полей, затем аудитор проверяет, что бизнес-логика и подтверждения не изменены, после этого исполнитель меняет шаблоны.
