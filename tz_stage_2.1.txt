## 2. Граница Этапа 2.1

Этап 2.1 относится только к **WB**.

Охват:

```text
2.1.1 Скачать по API цены WB,
      сформировать Excel-файл с ценами,
      дать его скачать и проверить,
      занести скачанные товары в справочник товаров конкретного магазина.

2.1.2 Скачать из календаря акций WB текущие акции,
      занести акции и состав товаров по акциям в БД,
      сформировать Excel-файлы акций,
      дать их скачать и проверить.

2.1.3 Выполнить расчёт скидок по данным 2.1.1 и 2.1.2,
      сформировать итоговый Excel-файл,
      пригодный для ручной загрузки в личном кабинете WB,
      дать его скачать и проверить.

2.1.4 Выполнить загрузку рассчитанных скидок в личный кабинет WB по API,
      с фиксацией upload ID, статусов, ошибок, частичных ошибок и audit/techlog.
```

Не входит:

```text
2.2 Ozon API;
автоматизация остатков;
автоматизация заказов;
автоматизация поставок;
производственный модуль;
закупки;
изменение бизнес-формул WB Excel Этапа 1;
удаление или замена Excel-режима;
изменение Stage 1 accepted comparison artifacts.
```

Excel-режим WB Этапа 1 остаётся штатным и резервным режимом. API-режим добавляется как новый режим работы, а не как замена существующего Excel-сценария.

## 3. Базовая продуктовая логика Этапа 2.1

Текущий ручной процесс WB:

```text
1. Пользователь скачивает из личного кабинета WB файл с ценами.
2. Пользователь скачивает из календаря акций WB файлы акций.
3. Система рассчитывает скидки.
4. Система записывает результат в Excel-файл с ценами.
5. Пользователь вручную загружает итоговый Excel в личный кабинет WB.
```

Этап 2.1 должен сохранить эту понятную модель, но заменить источники данных на API и добавить опциональную API-загрузку:

```text
2.1.1 API скачал цены
      ↓
      проверочный Excel-файл цен
      ↓
2.1.2 API скачал текущие акции
      ↓
      проверочные Excel-файлы акций
      ↓
2.1.3 расчёт скидок
      ↓
      итоговый Excel-файл для ручной загрузки в ЛК WB
      ↓
2.1.4 API-загрузка рассчитанных скидок в WB
```

Принципиальное ограничение:

```text
2.1.1 не меняет данные в WB.
2.1.2 не меняет данные в WB.
2.1.3 не меняет данные в WB.
2.1.4 — единственный подэтап, который отправляет изменения в WB через API.
```

## 6. Требования к 2.1.1 — получение цен WB по API

### 6.1. Назначение

Пользователь выбирает WB-магазин/кабинет и запускает действие:

```text
Скачать цены по API
```

Система должна:

```text
1. Проверить права пользователя.
2. Проверить доступ пользователя к магазину.
3. Проверить наличие активного WB API-подключения.
4. Получить товары и цены WB через API.
5. Сохранить snapshot полученных данных.
6. Сформировать Excel-файл цен.
7. Дать пользователю скачать Excel-файл.
8. Обновить справочник товаров конкретного магазина.
9. Создать operation/run/audit/techlog записи.
```

### 6.2. Основной API-метод

Для получения всех товаров с ценами использовать:

```text
GET /api/v2/list/goods/filter
```

Документация WB указывает, что для получения всех товаров нужно не задавать article-фильтр, использовать `limit=1000`, управлять смещением через `offset` и повторять запросы до пустого массива. Метод возвращает `nmID`, `vendorCode`, массив `sizes`, `currencyIsoCode4217`, `discount`, `clubDiscount`, `editableSizePrice`, `isBadTurnover`; в `sizes` есть `sizeID`, `price`, `discountedPrice`, `clubDiscountedPrice`, `techSizeName`. ([WB API][3])

Обязательные технические параметры:

```text
base URL:
  https://discounts-prices-api.wildberries.ru

endpoint:
  /api/v2/list/goods/filter

method:
  GET

auth:
  HeaderApiKey

pagination:
  limit = 1000
  offset = 0, 1000, 2000...
  stop when listGoods is empty

rate limit:
  10 requests / 6 seconds, interval 600 ms, burst 5
```

Официальная документация указывает лимит категории Prices and Discounts: 10 запросов за 6 секунд, интервал 600 ms, burst 5. ([WB API][3])

### 6.3. Excel-файл цен

Сформированный файл должен быть пригоден для проверки пользователем и для дальнейшего расчёта 2.1.3.

Минимальные обязательные колонки на основном листе:

```text
Артикул WB
Текущая цена
Новая скидка
```

Назначение колонок:

```text
Артикул WB
  nmID из WB API.

Текущая цена
  price из API.

Новая скидка
  пустая колонка, куда 2.1.3 запишет рассчитанную скидку.
```

Дополнительные диагностические поля допускаются только если они не ломают совместимость с текущим WB Excel-парсером. Если проектировщик хочет добавить дополнительные поля, предпочтительный вариант — отдельный диагностический лист, например:

```text
_api_raw
```

или отдельный detail report, а не изменение основной схемы файла.

### 6.4. Size-based pricing

WB API возвращает `sizes[]` и признак `editableSizePrice`; отдельный метод размеров применяется для товаров, где доступна установка цен по размерам. ([WB API][3])

Проектировщик должен явно описать поведение для товаров с несколькими размерами:

```text
1. Если у всех sizes одного nmID одинаковая price:
   использовать эту price как "Текущая цена".

2. Если sizes одного nmID имеют разные price:
   не подбирать цену молча;
   не использовать случайный первый размер;
   зафиксировать строку как спорную для расчёта;
   отразить случай в detail rows / warnings;
   не допускать API upload по такой строке до отдельного утверждённого правила.
```

Если проектировщик считает нужным добавить новый reason/result code для таких строк, он должен оформить это через документацию и ADR, так как Stage 1 WB reason/result codes были закрытым перечнем и расширяются только через утверждённое изменение. ([GitHub][7])

### 6.5. Обновление справочника товаров

Для каждого полученного `nmID` необходимо создать или обновить товар в справочнике товаров магазина.

Рекомендуемый маппинг:

```text
MarketplaceProduct.marketplace = "wb"
MarketplaceProduct.store = выбранный StoreAccount
MarketplaceProduct.sku = str(nmID)
MarketplaceProduct.external_ids = {
  "nmID": nmID,
  "vendorCode": vendorCode,
  "sizeIDs": [...],
  "techSizeNames": [...],
  "source": "wb_prices_api"
}
MarketplaceProduct.last_values = {
  "price": ...,
  "discount": ...,
  "discountedPrice": ...,
  "clubDiscount": ...,
  "clubDiscountedPrice": ...,
  "currencyIsoCode4217": ...,
  "editableSizePrice": ...,
  "isBadTurnover": ...
}
MarketplaceProduct.status = "active"
MarketplaceProduct.last_seen_at = operation timestamp
```

Если title не приходит из Prices API, не выдумывать название товара. Поле `title` оставить пустым или заполнить только из отдельного официально подключённого API-источника, если проектировщик включит его в спецификацию.

В текущей модели Stage 1 справочник товаров уже имеет `marketplace`, `store`, `external_ids`, `title`, `sku`, `barcode`, `status`, `last_values`, `first_detected_at`, `last_seen_at` и историю изменений. ([GitHub][8])

## 7. Требования к 2.1.2 — получение текущих акций WB

### 7.1. Назначение

Пользователь запускает действие:

```text
Скачать текущие акции WB
```

Система должна:

```text
1. Проверить права пользователя.
2. Проверить доступ к магазину.
3. Проверить активное WB API-подключение.
4. Получить список текущих акций из календаря WB.
5. Получить детали текущих акций.
6. Получить товары по каждой текущей акции, где это применимо.
7. Сохранить акции и строки товаров в БД.
8. Сформировать Excel-файл по каждой акции.
9. Дать пользователю скачать сформированные Excel-файлы.
10. Создать operation/run/audit/techlog записи.
```

### 7.2. Определение текущей акции

Проектировщик должен зафиксировать следующее определение:

```text
Текущая акция WB — акция, у которой:
  startDateTime <= now_utc < endDateTime
```

`now_utc` — текущий момент в UTC на момент запуска операции.

Нельзя использовать “ближайшие”, “будущие”, “доступные”, “последние” или “все акции” как замену текущих. Пользователь явно указал: **текущие, именно текущие**.

### 7.3. API-методы календаря акций

Использовать:

```text
GET /api/v1/calendar/promotions
GET /api/v1/calendar/promotions/details
GET /api/v1/calendar/promotions/nomenclatures
```

Метод списка акций возвращает список promotions с `id`, `name`, `startDateTime`, `endDateTime`, `type` и принимает `startDateTime`, `endDateTime`, `allPromo`, `limit`, `offset`. ([WB API][4])

Метод деталей принимает до 100 уникальных `promotionIDs` и возвращает детальную информацию, включая описание, преимущества, даты, counts, percentage, type и ranging. ([WB API][4])

Метод товаров акции возвращает товары, подходящие для участия в акции, и принимает `promotionID`, `inAction`, `limit`, `offset`; в ответе есть `id`, `inAction`, `price`, `currencyCode`, `planPrice`, `discount`, `planDiscount`. Метод не применяется к auto promotions. ([WB API][4])

### 7.4. Правило выборки текущих акций

Рекомендуемый технический алгоритм:

```text
1. Сформировать API window, который гарантированно покрывает now_utc:
   startDateTime = now_utc - 24h
   endDateTime   = now_utc + 24h

2. Запросить promotions list с allPromo=true.

3. Применить строгий локальный фильтр:
   startDateTime <= now_utc < endDateTime.

4. По отфильтрованным promotion IDs запросить details батчами до 100 IDs.

5. Для каждой regular promotion запросить nomenclatures:
   - inAction=true;
   - inAction=false;
   - с пагинацией limit=1000, offset до пустого массива.

6. Для auto promotion:
   - сохранить саму акцию и details;
   - сформировать запись о том, что список товаров через nomenclatures недоступен;
   - не выдумывать строки товаров;
   - отразить это в UI и detail report.
```

Если проектировщик считает, что `allPromo=false` лучше соответствует пользовательскому сценарию, он должен вынести это как функциональный вопрос заказчику через оркестратора. По умолчанию для “текущих акций” использовать `allPromo=true`, чтобы не потерять текущие акции, которые уже идут, но не классифицируются WB как “available for participating”.

## 8. Требования к 2.1.3 — расчёт скидок и итоговый Excel

### 8.1. Назначение

Пользователь запускает действие:

```text
Рассчитать скидки
```

Система должна:

```text
1. Взять последний или явно выбранный price export из 2.1.1.
2. Взять последний или явно выбранный набор current promotion exports из 2.1.2.
3. Применить существующую WB-логику расчёта скидок.
4. Сформировать итоговый Excel-файл.
5. Дать пользователю скачать итоговый Excel.
6. Не отправлять изменения в WB.
```

### 8.2. Бизнес-логика расчёта

Расчёт должен использовать ту же формулу и тот же порядок правил, что Stage 1 WB Excel:

```text
calculated_discount = ceil((1 - max_plan_price / current_price) * 100)

final_discount_pre_threshold = MIN(min_discount, calculated_discount)

1. Если артикул отсутствует во всех валидных файлах акций:
   final_discount = wb_fallback_no_promo_percent

2. Иначе если final_discount_pre_threshold > wb_threshold_percent:
   final_discount = wb_fallback_over_threshold_percent

3. Иначе:
   final_discount = final_discount_pre_threshold
```

Запрещено менять порядок правил, использовать float, обрезать результат вне диапазона 0..100 или частично обрабатывать workbook при ошибке диапазона. Stage 1 WB spec уже фиксирует decimal + ceil, закрытый набор WB reason/result codes и ошибку `wb_discount_out_of_range` при выходе за диапазон. ([GitHub][7])

### 8.3. Архитектурное требование

Проектировщик должен потребовать выделения общего расчётного ядра WB, если оно ещё не выделено.

Запрещено:

```text
- дублировать формулу отдельно для Excel mode и API mode;
- иметь две расходящиеся реализации WB расчёта;
- менять Stage 1 Excel-поведение;
- считать API-режим новым бизнес-правилом.
```

Правильная модель:

```text
WB calculation core
  ↑
  Excel input adapter
  API-generated Excel / API snapshot adapter
  ↓
WB output writer
```

### 8.4. Итоговый Excel

Итоговый Excel должен:

```text
- быть сформирован на основе price Excel из 2.1.1;
- записывать результат в колонку "Новая скидка";
- быть пригодным для ручной загрузки в личном кабинете WB;
- сохранять порядок строк;
- не разрушать формулы;
- не менять запрещённые колонки;
- иметь checksum;
- храниться как FileObject/FileVersion;
- быть связан с operation.
```

Если WB личный кабинет требует специальный формат, отличный от минимального Stage 1 price workbook, проектировщик должен зафиксировать точную схему файла. Нельзя оставлять “пригодный для загрузки” без формального определения.

### 8.5. Актуальность данных

Для 2.1.3 check/process актуальность должна учитывать:

```text
- выбранный store/account;
- выбранный price export operation/file version/snapshot;
- выбранный promotions export operation/file versions/snapshots;
- applied WB parameters;
- calculation logic version;
- current promotion filter timestamp;
- WB API data snapshot checksum.
```

Если пользователь перескачал цены или акции, старый расчёт не считается автоматически актуальным.

## 9. Требования к 2.1.4 — загрузка скидок WB по API

### 9.1. Назначение

Пользователь запускает действие:

```text
Загрузить скидки по API
```

Система должна:

```text
1. Проверить права пользователя.
2. Проверить object access к магазину.
3. Проверить активное WB API-подключение.
4. Проверить, что есть успешный расчёт 2.1.3.
5. Показать пользователю явное подтверждение API-загрузки.
6. Выполнить pre-upload drift check.
7. Отправить рассчитанные скидки в WB API.
8. Сохранить upload ID.
9. Проверить статус загрузки.
10. Получить детали ошибок при наличии.
11. Отразить результат в operation, detail rows, audit и techlog.
```

### 9.2. API-метод загрузки

Использовать:

```text
POST /api/v2/upload/task
```

WB API описывает этот метод как установку цен и скидок. В теле запроса передаётся `data`, максимум 1000 products, в каждом элементе `nmID`, `price`, `discount`; цена и скидка не могут быть одновременно пустыми. ([WB API][3])

Рекомендуемый payload для Stage 2.1.4:

```json
{
  "data": [
    {
      "nmID": 123,
      "discount": 30
    }
  ]
}
```

Проектировщик должен проверить и зафиксировать в спецификации, допускает ли WB API отправку только `discount` без `price`.

Рекомендуемое правило безопасности:

```text
Если API допускает discount-only payload:
  отправлять только nmID + discount,
  чтобы не перезаписать цену, которая могла измениться после расчёта.

Если API требует price:
  выполнять pre-upload drift check;
  брать price только из актуального WB API snapshot;
  при расхождении текущей API price с расчётным snapshot блокировать upload
  и требовать повторить 2.1.1–2.1.3.
```

Нельзя молча отправлять старую цену из расчётного Excel, если текущая цена в WB изменилась после скачивания.

### 9.3. Batching

Так как метод принимает максимум 1000 products, проектировщик должен описать batching:

```text
batch size <= 1000;
каждый batch имеет payload checksum;
каждый batch получает uploadID;
каждый uploadID хранится отдельно;
operation summary агрегирует batch statuses;
частичная ошибка одного batch не должна скрывать результат других batch.
```

### 9.4. Pre-upload drift check

Перед API-загрузкой система должна повторно получить по API данные по товарам, которые будут загружаться.

Для этого использовать:

```text
POST /api/v2/list/goods/filter
```

с `nmList`, батчами до 1000 nmID. Документация WB указывает, что этот метод возвращает данные по нескольким артикулам и принимает массив `nmList` от 1 до 1000 items. ([WB API][3])

Проверить:

```text
- товар существует;
- price не изменился относительно расчётного snapshot;
- товар не стал size-conflict;
- текущая скидка может быть обновлена;
- товар не исключён из загрузки по ошибкам расчёта.
```

Если обнаружен drift:

```text
API upload запрещён;
operation получает error;
UI показывает список товаров с расхождением;
пользователь должен повторить скачивание цен/акций и расчёт.
```

### 9.5. Проверка статуса загрузки

После отправки upload task нельзя считать операцию успешной по первому HTTP 200.

WB API указывает upload statuses:

```text
3 — processed, no errors, prices and discounts were updated;
4 — canceled;
5 — processed, but some products have errors; other products were updated;
6 — processed, but all products have errors;
1 — processing for promo calendar-related upload;
status 2 отсутствует.
```

Ошибки по товарам можно получить через processed upload details или unprocessed upload details. ([WB API][3])

Проектировщик должен описать polling/status check contract:

```text
1. После POST сохранить uploadID.
2. Проверять processed upload state:
   GET /api/v2/history/tasks?uploadID=...
3. Если upload в processing/unprocessed state:
   проверять:
   GET /api/v2/buffer/tasks?uploadID=...
4. При ошибках товаров получать details:
   GET /api/v2/history/goods/task
   GET /api/v2/buffer/goods/task
5. Сохранять статусы и errorText в безопасный snapshot.
```

### 9.6. Mapping статусов в Operation

Рекомендуемый mapping:

```text
WB status 3:
  Operation.status = completed_success

WB status 5:
  Operation.status = completed_with_warnings
  UI message = "Частично загружено: часть товаров обновлена, часть с ошибками"
  detail rows фиксируют success/error по каждому товару

WB status 6:
  Operation.status = completed_with_error
  UI message = "Загрузка не применена по всем товарам из-за ошибок"

WB status 4:
  Operation.status = completed_with_error
  UI message = "Загрузка отменена WB"

HTTP/API auth/rate/timeout/application failure:
  Operation.status = interrupted_failed или completed_with_error
  в зависимости от того, был ли uploadID создан

WB 208 already exists:
  не отправлять повторно payload вслепую;
  сохранить факт duplicate/alreadyExists;
  попытаться получить статус существующего uploadID, если он доступен;
  если uploadID не получен, показать пользователю безопасную ошибку и зафиксировать techlog.
```

### 9.7. Quarantine

WB API указывает, что если новая цена со скидкой минимум в 3 раза меньше старой, товар может попасть в карантин цены, и ошибка будет доступна в методах статуса загрузки; отдельный метод quarantine goods возвращает информацию о товарах в карантине. ([WB API][3])

Проектировщик должен описать:

```text
- как выявлять quarantine-related errors;
- как отображать их в UI;
- какой reason/result code использовать;
- как отражать это в detail rows;
- как не скрывать частично применённые изменения.
```

## 15. Acceptance criteria Stage 2.1

### 15.1. 2.1.1 accepted, если

```text
- пользователь с правами может скачать цены WB по API;

- history товара создана;
- operation создана;
- audit создан;
- techlog создаётся при ошибках;
- секреты не раскрыты;
- rate limits соблюдаются;
- size-conflict cases не обрабатываются молча.
```

### 15.2. 2.1.2 accepted, если

```text
- система скачивает именно текущие акции по правилу start <= now < end;
- детали акций сохраняются;
- regular promotions получают товары через nomenclatures;
- auto promotions не ломают операцию, а фиксируются как ограниченные;
- сформированы Excel promo files с обязательными колонками;
- Excel files можно скачать;
- данные акций сохранены в БД;
- operation/audit/techlog работают;
- allPromo/window/current-filter documented and tested.
```

### 15.3. 2.1.3 accepted, если

```text
- расчёт использует ту же WB-логику, что Stage 1;
- формула и порядок правил не изменены;
- используются decimal + ceil;
- итоговый Excel записывает "Новая скидка";
- итоговый Excel пригоден для ручной загрузки в ЛК WB;
- output file скачивается;
- detail rows показывают причины;
- ошибки блокируют дальнейший upload;
- snapshot параметров сохранён;
- повторный расчёт создаёт новую operation/file version.
```

### 15.4. 2.1.4 accepted, если

```text
- API upload возможен только после успешного расчёта;
- есть explicit confirmation;
- pre-upload drift check выполняется;
- drift блокирует upload;
- payload бьётся на batch <= 1000 товаров;
- uploadID сохраняется;
- status polling выполняется;
- statuses 3/4/5/6 корректно маппятся;
- partial errors отображаются;
- quarantine errors отображаются;
- 208 already exists обработан безопасно;
- 429/rate limit обработан через backoff;
- API failures не раскрывают секреты;
- audit и techlog созданы;
- operation immutable после завершения.
```

## 17. Обязательные GAP/ADR решения

Проектировщик должен проверить, нужны ли новые ADR/GAP.

Минимально ожидаемые ADR:

```text
ADR-0016: Stage 2 split — 2.1 WB API, 2.2 Ozon API.
ADR-0017: WB API source-to-Excel flow before API upload.
ADR-0018: WB current promotions definition.
ADR-0019: WB API upload safety: confirmation + drift check.
ADR-0020: WB API reason/result codes.
```

Если проектировщик обнаружит спорный функциональный вопрос, он должен оформить GAP и передать через оркестратора заказчику.

Потенциальные GAP-кандидаты:

```text
- что делать с WB товарами, у которых разные цены по размерам;
- включать ли в расчёт auto promotions, если WB API не отдаёт nomenclatures;
- нужен ли пользователю единый zip всех promo Excel-файлов или отдельное скачивание по акции;
- разрешать ли загрузку только части товаров при partial errors;
- считать ли товары с price drift абсолютным блокером upload или подтверждаемым warning.
```

Для текущего проектирования рекомендуемые решения такие:

```text
size price conflict -> блокировать строку для upload до отдельного правила;
auto promotion without nomenclatures -> сохранить акцию, не создавать выдуманные строки;
promo files -> поддержать отдельные файлы, zip/package как optional enhancement;
partial errors -> фиксировать как completed_with_warnings, потому что часть товаров уже обновлена WB;
price drift -> блокер upload, требуется пересчёт.
```
