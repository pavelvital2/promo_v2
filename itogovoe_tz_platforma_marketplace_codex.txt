# 2. Цель продукта и бизнес-контекст

## 2.1. Исходная ситуация

В настоящее время используется рабочая программа расчёта скидок для Wildberries и Ozon через Excel-файлы:

- менеджеры выгружают Excel-файлы из личных кабинетов маркетплейсов;
- обрабатывают их в программе;
- загружают результат обратно в маркетплейс.

Также существует отдельный модуль / проект WB API.

## 2.2. Цель создания новой платформы

Цель проекта - создать новую расширяемую веб-платформу для управления операциями маркетплейсов и дальнейшей автоматизации внутренних процессов.

Платформа должна поэтапно закрывать:

- скидки;
- цены;
- остатки;
- продажи;
- среднесуточные продажи;
- расчёт потребности;
- задания в работу;
- поставки;
- отгрузки;
- производство;
- производственные журналы по операторам;
- закупки;
- другие операционные и аналитические процессы.

## 2.3. Главный принцип продукта

Новая система проектируется как отдельный новый продукт. Она не является копией старой программы, но должна воспроизвести и формально зафиксировать прикладную бизнес-логику расчёта скидок WB/Ozon, которая необходима для замены текущего Excel-сценария.

## 2.4. Базовая единица модульности

Базовая единица платформы - бизнес-функция, а не формат интеграции.

К бизнес-функциям относятся:

- Скидки;
- Цены;
- Остатки;
- Продажи;
- Поставки;
- Производство;
- Закупки;
- иные будущие направления.

Внутри бизнес-функции допускаются режимы:

- Excel;
- API;
- ручной;
- сервисный;
- автоматический;
- иные будущие режимы.

Не допускается строить архитектуру вокруг Excel или API как главной сущности. Excel и API являются режимами работы бизнес-модулей.

## 2.5. Роль Excel

Excel является постоянным штатным и резервным режимом работы модулей. После появления API Excel-режим не удаляется из архитектуры.

Каждый будущий модуль, где это применимо, должен иметь возможность работать:

- с Excel-файлами на загрузку и выгрузку;
- через API;
- с сохранением единой модели операций, аудита, файлов, прав и объяснимости результата.

# 3. Этапы развития и границы этапа 1

## 3.1. Обязательный объём этапа 1

Этап 1 должен реализовать:

- платформенный каркас;
- веб-панель;
- пользователей, роли, права, доступы и объектные ограничения;
- модель магазинов / кабинетов;
- файловый контур;
- операции, проверки, обработки, run-контекст;
- аудит действий;
- технический журнал;
- базовые экспорты;
- модуль «Скидки»;
- режим Excel модуля «Скидки» для Wildberries;
- режим Excel модуля «Скидки» для Ozon;
- настройки расчёта WB;
- приёмочный контур по контрольным файлам WB/Ozon;
- эксплуатационный минимум: резервное копирование, восстановление, обновление, системные уведомления.

Главный практический результат этапа 1: платформа должна быть пригодна для реальной замены текущей рабочей программы по сценарию расчёта скидок WB и Ozon через Excel.

## 3.2. Что не входит в обязательную прикладную реализацию этапа 1

В этап 1 не входит подробная реализация следующих прикладных модулей:

- расчёт скидок по API;
- загрузка скидок в личные кабинеты через API;
- редактирование цен;
- остатки;
- продажи;
- расчёт среднесуточных продаж;
- расчёт задания в работу;
- производство;
- производственные журналы;
- поставки;
- отгрузки;
- закупки.

Эти направления должны быть заложены архитектурно, но не должны детально проектироваться в составе этапа 1, если это не влияет на базовую архитектуру или текущий функционал.

## 3.3. Ближайшая целевая последовательность развития

Платформа должна быть совместима со следующей последовательностью развития:

1. Этап 1 - расчёт скидок WB/Ozon по Excel.
2. Этап 2 - подключение API, расчёт скидок по данным API и загрузка полученных скидок в личные кабинеты WB/Ozon.
3. Этап 3 - редактирование цен на шевроны по Excel.
4. Этап 4 - редактирование цен по API.
5. Этап 5 - вычисление остатков, расчёт продаж за необходимый период и расчёт задания в работу.
6. Последующие этапы - производство, поставки, журналы операторов, закупки и другие процессы.

## 3.4. Будущая сквозная цепочка

Архитектура этапа 1 должна быть совместима со следующей будущей цепочкой:

- загрузка данных по остаткам через Excel или API;
- загрузка данных по продажам за указанный период через Excel или API;
- расчёт среднесуточных продаж;
- расчёт потребности на указанный горизонт;
- формирование списка в работу;
- передача в производство;
- распределение по операторам;
- фиксация изготовления;
- передача в поставку;
- отгрузка.

Разделы «Производство», «Поставка / Отгрузка» и иные будущие разделы не обязаны отображаться в интерфейсе этапа 1 как рабочие области, пока в них нет реализованного прикладного функционала.

# 4. Архитектурный стиль, контур развёртывания и база данных

## 4.1. Архитектурный стиль старта

Платформа проектируется как модульный монолит:

- единое приложение;
- единая база данных;
- единая авторизация;
- единый UI;
- единая история операций;
- единый аудит;
- единый технический журнал;
- единый execution layer;
- единый файловый контур.

## 4.2. Границы модулей

Несмотря на монолитный старт, архитектура должна иметь:

- жёсткие границы модулей;
- формализованные интерфейсы между модулями;
- запрет на хаотичные прямые зависимости;
- готовность к будущему выделению отдельных сервисов без переписывания доменного ядра.

## 4.3. Единая система

На этапе 1 система является единой общей платформой:

- одна веб-панель;
- одна общая PostgreSQL база данных;
- одно приложение;
- единый эксплуатационный контур.

Разделение доступа к данным выполняется ролями, правами и объектными ограничениями по магазинам / кабинетам. Выделение отдельных инсталляций или отдельных изолированных контуров на этапе 1 не требуется.

## 4.4. Production baseline

Базовый production-вариант развёртывания этапа 1:

- Ubuntu VPS;
- PostgreSQL;
- nginx;
- systemd;
- веб-доступ через браузер;
- web UI + HTTP API;
- без Docker как обязательного условия production baseline.

Иные варианты развёртывания допустимы в будущем, но не входят в обязательный объём этапа 1.

## 4.5. Доступ к веб-панели

Пользовательский доступ к веб-панели на этапе 1 осуществляется через обычный веб-интерфейс в браузере по адресу системы с аутентификацией по логину и паролю.

Использование отдельного пользовательского VPN как обязательного условия входа в систему на этапе 1 не требуется.

## 4.6. Test / staging

Отдельная test/staging-среда на этапе 1 не является обязательным условием ввода системы в эксплуатацию. При этом архитектура, документация и порядок поставки должны предусматривать возможность последующего развёртывания отдельного тестового контура без переработки базовой схемы системы.

## 4.7. База данных

На этапе 1 используется одна общая PostgreSQL база данных для всех прикладных и системных сущностей платформы.

Разделение на отдельные базы или отдельные постоянные контуры хранения на этапе 1 не требуется.

Архитектура модели данных, схема модулей и правила миграций должны предусматривать дальнейшее расширение системы без ломки базовой структуры и без необходимости перепроектирования уже реализованных сущностей.

# 6. Правила описания экранов и обязательный перечень экранов

## 6.1. Что считается экраном

Экран - это не только отдельная веб-страница, но и самостоятельное пользовательское экранное состояние / маршрут, имеющий:

- отдельное назначение;
- отдельный набор данных;
- отдельный набор действий;
- отдельные статусы, сообщения, ошибки и предупреждения;
- отдельную роль в пользовательском сценарии.

Одна техническая страница может содержать несколько экранных состояний, но если состояния различаются по назначению, данным, действиям, правам или сценарию, в проектной документации они описываются как разные экраны.

## 6.2. Формат описания каждого экрана

Исполнительная проектная документация по каждому экрану должна содержать:

- назначение;
- раздел интерфейса;
- роли и права, имеющие доступ;
- входные точки;
- отображаемые данные;
- доступные действия;
- обязательные элементы управления;
- фильтры, поиск, сортировку, пагинацию, если применимо;
- сообщения, ошибки, предупреждения и статусы;
- переходы на другие экраны;
- связь со сценариями;
- критерии готовности экрана.

## 7.2. Будущие сущности, которые должны быть заложены архитектурно

Архитектурно должны быть предусмотрены будущие сущности:

- внутренняя карточка номенклатуры;
- модель товара;
- вариация товара;
- предложение маркетплейса;
- остаток;
- продажа;
- среднесуточная продажа;
- потребность;
- задание в работу;
- задание в производство;
- производственный маршрут;
- оператор;
- производственная операция;
- факт изготовления;
- партия;
- поставка;
- отгрузка;
- склад;
- состояние движения товара;
- закупка.

# 8. Модель магазинов, кабинетов и подключений

## 8.1. Двухуровневая модель

В системе должна быть двухуровневая модель:

- верхний уровень - группа / бренд / бизнес-направление;
- нижний уровень - конкретный магазин / кабинет маркетплейса.

На этапе 1 основной рабочей сущностью является конкретный магазин / кабинет маркетплейса.

## 8.2. Привязки магазина / кабинета

К магазину / кабинету привязываются:

- операции;
- файлы;
- параметры;
- права пользователей;
- история изменений;
- блок будущего API-подключения;
- маркетплейс-товары;
- будущие API-сценарии;
- будущие цены, остатки, продажи и поставки.

## 8.3. Карточка магазина / кабинета

На этапе 1 должна быть реализована полноценная карточка магазина / кабинета.

Карточка содержит:

- видимый идентификатор;
- название;
- группу / бренд / бизнес-направление;
- маркетплейс;
- тип кабинета;
- статус;
- параметры этапа 1;
- блок будущего API-подключения;
- доступы пользователей;
- историю изменений;
- связанные операции;
- связанные товары;
- служебные комментарии.

Карточка магазина / кабинета является основной рабочей сущностью для параметров, доступов, подключений, истории изменений и просмотра связанных операций.

## 8.4. История изменений магазина

В карточке магазина / кабинета должна быть история изменений параметров и настроек с указанием:

- даты и времени;
- пользователя;
- изменённого поля;
- старого значения;
- нового значения;
- источника изменения, если применимо.

## 8.5. Блок API-подключения этапа 1

На этапе 1 в карточке магазина / кабинета реализуется защищённый блок API-подключения.

В нём можно хранить API-ключи и параметры будущих интеграций, но API-сценарии расчёта и загрузки скидок не входят в этап 1.

Блок должен быть помечен для пользователя как: «подготовлено для этапа 2, в этапе 1 не используется».

Для блока API должны существовать отдельные права:

- просмотр блока подключения;
- изменение блока подключения;
- добавление / изменение API-ключей и параметров подключения.

Архитектура блока подключений должна предусматривать возможность наличия у одного магазина нескольких самостоятельных подключений и интеграционных параметров для разных модулей и будущих сценариев.

# 9. Модель товаров и внутренней номенклатуры

## 9.1. Общий принцип

Система должна поддерживать внутреннюю карточку изделия / номенклатуры. Карточки и предложения WB/Ozon привязываются к внутреннему изделию.

Один внутренний товар может быть связан с несколькими карточками WB/Ozon и с несколькими магазинами / кабинетами.

Цены, скидки, акции, остатки, настройки участия в акциях и результаты операций хранятся не на уровне внутреннего изделия в целом, а на уровне конкретного маркетплейса, магазина / кабинета и связанного маркетплейс-предложения.

Один и тот же внутренний товар может иметь разные цены, скидки и акции в разных магазинах, включая разные магазины одного маркетплейса.

## 9.2. Двухуровневая товарная модель

Архитектурно должна быть предусмотрена двухуровневая товарная модель:

- верхний уровень - модель товара;
- нижний уровень - конкретная вариация, по которой считаются продажи, остатки, производство и поставки.

Пример:

- модель: «Футболка с вышивкой “Кот”»;
- вариация: «белая / M»;
- вариация: «белая / L»;
- вариация: «чёрная / M».

На этапе 1 полная внутренняя номенклатура не реализуется как рабочий производственный справочник, но должна быть заложена архитектурно.

## 9.3. Маркетплейс-товар этапа 1

На этапе 1 реализуется карточка маркетплейс-товара, привязанная к конкретному маркетплейсу и конкретному магазину / кабинету.

Товары создаются и обновляются автоматически на основании валидных строк загружаемых Excel-файлов WB/Ozon.

Отсутствие товара в справочнике до загрузки файла не является ошибкой.

Карточка маркетплейс-товара этапа 1 должна хранить:

- маркетплейс;
- магазин / кабинет;
- идентификаторы товара из Excel;
- название товара, если оно есть в файле;
- артикул / SKU / штрихкод, если есть в файле;
- последние известные значения из обработанных файлов;
- связанные проверки и обработки;
- связанные входные и выходные файлы;
- статус товара;
- историю появления / обновления;
- блок будущей связи с внутренней номенклатурой.

# 10. Производство и склад как будущие архитектурные направления

## 10.1. Производственный маршрут

Для внутренних вариаций товара архитектура должна поддерживать производственный маршрут: набор операций, необходимых для изготовления / подготовки товара.

Минимально обязательная будущая логика производства:

- задания в производство;
- фиксация факта по операциям;
- исполнитель;
- дата / время;
- количество;
- статус выполнения.

Архитектура должна позволять позже добавить материалы, нормы, полуфабрикаты и полную спецификацию изделия без переделки товарной модели.

## 10.2. Задания / партии

Производственный учёт в будущих этапах должен строиться через задания / партии.

Задание содержит:

- товарные вариации;
- операцию;
- плановое количество;
- исполнителя или группу исполнителей;
- статус;
- историю выполнения.

Сотрудник фиксирует частичное или полное выполнение по заданию.

Архитектура должна позволять в будущем добавить поштучный учёт, сканирование и индивидуальные коды изделий.

## 10.3. Складская модель

Архитектурная цель складской модели - полноценный учёт:

- внутреннего склада;
- остатков на маркетплейсах;
- товаров в производстве;
- товаров в поставке / отгрузке;
- транзита;
- нескольких складов в будущем.

Начать можно с упрощённой реализации: один внутренний склад плюс отдельный учёт остатков маркетплейсов. При этом модель данных должна позволять расширение до нескольких складов и состояний движения товара.

Подробные требования к производству, складам и поставкам не входят в обязательную прикладную спецификацию этапа 1, если не влияют на платформенный каркас.

# 11. Пользователи, роли, права и доступы

## 11.1. Общая модель доступа

В системе должны быть разделены:

- права на действия;
- доступы к разделам / вкладкам;
- объектные доступы к конкретным магазинам / кабинетам.

Права и доступы не являются одним и тем же.

Примеры прав:

- создать магазин;
- изменить параметры расчёта скидок;
- подключить API-ключ;
- запустить проверку;
- запустить обработку;
- подтвердить предупреждения;
- скачать итоговый файл.

Примеры доступов:

- доступ к просмотру логов;
- доступ к просмотру операций;
- доступ к вкладке «Цены»;
- доступ к вкладке «Производство»;
- доступ к разделу «Администрирование».

## 11.2. Роль как шаблон

Роль является шаблоном доступа.

В роли заранее определяются:

- доступные действия;
- доступные разделы;
- допустимые области доступа;
- иные связанные права, предусмотренные системой.

Владелец или администратор создаёт шаблон роли, например «Менеджер», выбирает права и доступы этой роли и назначает роль сотрудникам.

Пользователю назначается одна основная роль-шаблон.

## 11.3. Индивидуальные настройки пользователя

Поверх роли пользователю могут быть выданы индивидуальные дополнительные права и доступы.

Индивидуальные настройки пользователя могут не только добавлять права, но и запрещать отдельные права, выданные ролью.

Индивидуальные разрешения и запреты могут задаваться по конкретному магазину / кабинету.

При конфликте разрешения и прямого запрета приоритет имеет прямой запрет на уровне пользователя или конкретного магазина / кабинета.

## 11.4. Доступы к магазинам / кабинетам

Доступ к магазину / кабинету задаётся отдельно и применяется поверх роли.

Если у пользователя нет доступа к конкретному магазину / кабинету, он не должен видеть связанные с ним операции, файлы, товары, параметры и аудит, кроме случаев глобальных административных прав.

Права пользователя применяются только в пределах доступных ему магазинов / кабинетов, если право связано с магазином / кабинетом.

## 11.5. Владелец

В системе должна быть системная роль «Владелец».

Владелец имеет полный доступ ко всем разделам, магазинам / кабинетам, операциям, настройкам, пользователям, ролям, правам, аудиту, техжурналу и подключениям.

Владелец не может быть ограничен по магазинам, удалён, заблокирован или лишён критичных прав администратором.

Роль «Владелец» является верхним аварийно-управленческим уровнем доступа и предназначена для сохранения конечного контроля заказчика над системой.

## 11.6. Администраторы

Администратор является делегируемой ролью и может быть ограничен правами, доступами и магазинами / кабинетами.

Административные права должны поддерживать два уровня:

- локальный администратор - управляет пользователями, доступами и настройками только в пределах доступных ему магазинов / кабинетов;
- глобальный администратор - может управлять всеми пользователями, ролями, правами, магазинами и кабинетами, но не может ограничивать, блокировать, удалять или менять критичные права владельца.

## 11.7. Рабочая модель пользователя этапа 1

На этапе 1 пользователь должен иметь как минимум:

- логин;
- пароль / безопасно хранимый парольный секрет;
- отображаемое имя;
- статус активности;
- назначенную основную роль;
- объектные доступы к магазинам / кабинетам;
- индивидуальные разрешения;
- индивидуальные запреты;
- историю значимых изменений;
- историю блокировок.

Дополнительные кадровые и организационные поля не являются обязательными для этапа 1, но архитектура должна допускать их последующее добавление без переработки базовой модели.

## 11.8. Аутентификация

На этапе 1 вход в веб-панель осуществляется по внутренней учётной записи пользователя:

- логин;
- пароль.

Архитектура аутентификации должна предусматривать возможность последующего добавления иных способов входа без переработки базовой модели пользователей.

## 11.9. Типовые роли этапа 1 и будущих этапов

На этапе 1 должны быть реализуемы как минимум следующие типовые роли / шаблоны:

- Владелец;
- глобальный администратор;
- локальный администратор;
- менеджер маркетплейсов;
- наблюдатель / просмотр без изменений.

Роли производства, склада, упаковщиков и операторов вышивки не являются обязательными рабочими ролями этапа 1, но архитектура прав должна позволять добавить их в будущих этапах.

## 11.10. Обязательные права по Excel-сценариям

Права по сценариям WB -> Скидки -> Excel и Ozon -> Скидки -> Excel должны быть разделимы как самостоятельные наборы, даже если структура действий совпадает.

Минимально обязательные права:

- видеть сценарий;
- загружать входной файл;
- запускать проверку;
- просматривать результат проверки;
- просматривать детализацию ошибок / предупреждений;
- подтверждать предупреждения;
- запускать обработку;
- просматривать результат обработки;
- скачивать выходной файл;
- скачивать отчёт детализации;
- просматривать связанные операции;
- запускать повторную проверку;
- запускать повторную обработку.

## 11.11. Права по параметрам и настройкам

Минимально обязательные права:

- просмотр системных параметров;
- изменение системных параметров;
- просмотр параметров магазина / кабинета;
- изменение параметров магазина / кабинета;
- просмотр истории изменений параметров;
- просмотр источника применённого значения параметра;
- просмотр служебных настроек, если они есть;
- изменение служебных настроек, если они есть.

## 11.12. Права по магазинам / кабинетам / подключениям

Минимально обязательные права:

- просмотр списка магазинов;
- просмотр карточки магазина;
- создание магазина;
- изменение магазина;
- изменение параметров магазина;
- просмотр блока подключения / интеграции;
- изменение блока подключения / интеграции;
- добавление и изменение API-ключа / параметров подключения;
- назначение и изменение ограничений доступа к магазину;
- просмотр связанных операций магазина;
- просмотр истории изменений магазина.

## 11.13. Права по аудиту и техническим журналам

Минимально обязательные права:

- просмотр списка аудита;
- просмотр карточки записи аудита;
- просмотр списка техжурнала;
- просмотр карточки записи техжурнала;
- право видеть ограниченный контур записей;
- право видеть полный контур записей;
- право видеть чувствительные технические детали только уполномоченным ролям.

# 12. Операции, run, проверка и обработка

## 12.1. Раздел «Операции»

Раздел «Операции» является самостоятельным рабочим реестром пользовательских, автоматических и сервисных бизнес-операций системы.

Раздел не подменяет аудит действий и технический журнал.

Основной способ работы пользователя на этапе 1 - через сценарии модулей маркетплейсов. Раздел «Операции» используется как:

- журнал;
- контроль;
- поиск;
- история;
- инструмент разбора проблемных случаев.

## 12.2. Операция как неизменяемая запись

После завершения операция и её зафиксированный результат считаются неизменяемыми.

Не допускаются:

- изменение результата;
- подмена связанных файлов;
- изменение применённых параметров;
- корректировка итоговых данных завершённой операции.

Для повторного действия, исправления или повторной попытки создаётся новая самостоятельная операция.

## 12.3. Обязательные поля операции

Каждая операция должна иметь:

- уникальный видимый пользователю идентификатор;
- маркетплейс;
- модуль;
- режим: Excel / будущий API / иной режим;
- магазин / кабинет;
- тип операции: проверка / обработка / будущие типы;
- статус;
- инициатора операции;
- контекст выполнения / от чьего имени она выполнялась;
- способ запуска: ручной, автоматический, сервисный, API, будущий способ;
- дату и время запуска;
- дату и время завершения;
- входные файлы и их версии;
- выходной файл, если есть;
- проверку-основание для обработки;
- применённые параметры;
- версию бизнес-логики;
- краткий итог;
- ошибки и предупреждения;
- детализацию по строкам;
- подтверждение предупреждений: кто, когда, что подтвердил;
- ссылки на аудит и технический журнал при наличии проблем.

## 12.4. Снимок параметров и версия бизнес-логики

Каждая завершённая операция должна хранить снимок фактически применённых параметров и их источников на момент выполнения.

Каждая операция должна хранить идентификатор или версию применённой бизнес-логики / набора правил обработки.

## 12.5. Раздельность проверки и обработки

Операции «Проверка» и «Обработка» являются самостоятельными типами операций.

Каждая имеет:

- собственный тип;
- собственный статус;
- собственную запись в истории;
- собственный результат;
- собственные сообщения;
- собственную карточку.

Обработка не является независимой от проверки. Система обязана хранить:

- на основании какой проверки была запущена обработка;
- требовалось ли подтверждение предупреждений;
- кем и когда оно было подтверждено.

## 12.6. Пользовательский сценарий «Проверить / Обработать»

Пользователь может сразу нажать либо «Проверить», либо «Обработать».

Если нажата кнопка «Проверить», система выполняет только проверку, создаёт операцию проверки, показывает результат проверки и не создаёт итоговый файл.

Если нажата кнопка «Обработать», система должна выполнить обработку по допустимой проверке-основанию.

Если для текущего набора файлов уже есть актуальная успешная проверка, повторная проверка не выполняется. Обработка создаётся на основании этой проверки.

Если актуальной успешной проверки нет, система автоматически выполняет проверку и, если ошибок нет, сразу продолжает обработку.

Проверка и обработка остаются отдельными операциями. Обработка всегда должна хранить ссылку на проверку, на основании которой она была выполнена.

## 12.7. Актуальность проверки

Проверка считается актуальной для обработки только если совпадают:

- маркетплейс;
- магазин / кабинет;
- набор входных файлов;
- конкретные версии файлов;
- применяемые параметры;
- версия бизнес-логики.

Если хотя бы один из этих элементов изменился, при нажатии «Обработать» система должна выполнить новую проверку.

Это правило заменяет прежнюю формулировку, по которой при «Обработать» всегда выполнялась новая проверка.

## 12.8. Допустимая основа для обработки

Обработка может быть запущена только по результату проверки без ошибок:

- если есть ошибки - обработка запрещена;
- если ошибок нет, но есть подтверждаемые предупреждения - обработка разрешается только после явного подтверждения предупреждений;
- если нет ошибок и нет подтверждаемых предупреждений - обработка запускается без дополнительного подтверждения.

## 12.9. Предупреждения

Предупреждения делятся на:

- информационные;
- подтверждаемые.

Информационные предупреждения не требуют действий пользователя и не блокируют обработку.

Подтверждаемые предупреждения требуют явного подтверждения перед обработкой.

Если подтверждаемые предупреждения найдены при сценарии «Обработать», система должна показать экран подтверждения и не запускать обработку до действия пользователя.

После подтверждения пользователь нажимает «Подтвердить и обработать», и обработка выполняется на основании этой проверки.

Факт подтверждения должен сохраняться:

- кто подтвердил;
- когда подтвердил;
- по какой проверке;
- для какой операции обработки;
- какие предупреждения были подтверждены.

## 12.10. Повторная проверка и повторная обработка

Система должна разрешать повторную проверку. Каждая повторная проверка:

- создаёт новую самостоятельную операцию;
- не перезаписывает предыдущую;
- имеет собственную историю и собственный результат.

Система должна разрешать повторную обработку. Каждая повторная обработка:

- создаёт новую самостоятельную операцию;
- не перезаписывает предыдущую;
- имеет собственную историю;
- имеет собственный выходной файл, если он сформирован;
- хранит связь с конкретной проверкой-основанием.

Если допустимых проверок несколько, система не должна скрыто выбирать проверку автоматически. Пользователь должен понимать, какая именно проверка является основанием обработки.

## 12.11. Run и черновой контекст

Система должна поддерживать внутреннюю контейнерную сущность пользовательского запуска - run.

Run объединяет:

- связанные файлы;
- версии файлов;
- проверки;
- обработки;
- результаты.

В пользовательском интерфейсе основной рабочей сущностью остаётся операция. Run используется как внутренняя контейнерная сущность и не обязан быть главным пользовательским объектом этапа 1.

На этапе 1 система должна поддерживать временный черновой контекст текущего пользовательского запуска в рамках активного сценария, позволяющий:

- загружать файлы;
- заменять файлы;
- удалять файлы;
- формировать набор файлов до запуска операции.

Обязательное сохранение полноценного постоянного черновика между отдельными входами пользователя в систему на этапе 1 не требуется.

## 12.12. Статусы операций

Статусы, типы операций, способы запуска, уровни сообщений и reason/result codes должны быть фиксированными системными справочниками.

Пользователь не редактирует эти коды.

Коды используются единообразно в:

- бизнес-логике;
- хранении данных;
- интерфейсе;
- экспортах;
- аудите;
- тестировании.

Минимально допустимые статусы проверки:

- создана / принята;
- выполняется;
- завершена без ошибок;
- завершена с предупреждениями;
- завершена с ошибками;
- прервана / сбойная.

Минимально допустимые статусы обработки:

- создана / принята;
- выполняется;
- завершена успешно;
- завершена с предупреждениями;
- завершена с ошибкой;
- прервана / сбойная.

На этапе 1 ручная отмена операции пользователем не реализуется.

Если операция прервана из-за сбоя приложения, сервера, базы данных или файлового хранилища, система переводит её в статус прерванной / сбойной операции.

Автоматическое продолжение такой операции не выполняется. Для повторной попытки пользователь запускает новую операцию.

Архитектура должна позволять добавить ручную отмену или безопасное возобновление в будущих этапах.

# 13. Файловый контур и хранение

## 13.1. Файл как сущность

Файл является самостоятельной сущностью системы с:

- метаданными;
- сроком хранения;
- версионностью;
- связями с операциями.

В пользовательском интерфейсе основной рабочей сущностью должна оставаться операция, а не самостоятельный файловый реестр.

## 13.2. Хранение файлов

На этапе 1 хранятся:

- входные файлы;
- выходные файлы;
- метаданные файлов;
- связи файлов с операциями и версиями.

Физические входные и выходные файлы хранятся в серверном файловом хранилище системы.

В PostgreSQL хранятся:

- метаданные файлов;
- сведения о версиях;
- сроки хранения;
- связи с операциями;
- идентификаторы / пути хранения.

## 13.3. Версии файлов

Каждая операция должна иметь явную и неизменяемую связь с конкретной версией входного файла.

Повторная загрузка исправленного файла, даже с тем же именем, создаёт новую версию файла и не изменяет связи уже завершённых операций с ранее использованной версией.

Если пользователь исправил файл и загрузил его заново:

- это считается новой версией входного файла;
- версия должна быть связана с прежними версиями в рамках общего рабочего контекста;
- каждая операция должна быть привязана к конкретной версии входного файла.

Обработка старой версии файла допустима только если:

- у старой версии есть допустимая проверка;
- пользователь явно выбирает именно эту версию и именно эту проверку;
- система явно показывает, что пользователь запускает обработку не по последней версии.

## 13.4. Сроки хранения

Файлы хранятся 3 дня.

История и метаданные операций хранятся 90 дней, если в исполнительной документации не утверждён более длительный срок.

Если срок хранения файла истёк:

- запись об операции не удаляется;
- в истории видно, что файл был связан с операцией;
- видно, что срок хранения файла истёк;
- связь операции с файловой сущностью сохраняется как историческая запись.

# 14. Параметры и настройки расчёта

## 14.1. Общий принцип

Параметры расчёта относятся к конкретному магазину / кабинету, если не указано иное.

Системные значения используются как значения по умолчанию.

Для каждой операции система должна сохранять:

- фактически применённое значение параметра;
- источник значения: системное или магазинное;
- снимок параметров на момент выполнения.

Изменение параметров влияет только на новые операции. Прошлые операции сохраняют снимок фактически применённых значений.

## 14.2. Параметры WB на рабочем экране

На экране «Маркетплейсы -> WB -> Скидки -> Excel» должен быть блок параметров расчёта WB:

- порог скидки;
- скидка при превышении порога;
- скидка при отсутствии товара в акциях.

Эти параметры относятся к выбранному магазину / кабинету, а не к отдельному файлу.

На этом экране параметры можно смотреть и менять, если у пользователя есть право «изменение параметров магазина / кабинета».

Изменения сохраняются в настройках выбранного магазина / кабинета, попадают в историю изменений и применяются к новым проверкам / обработкам.

В операции сохраняется снимок применённых параметров.

Для «Ozon -> Скидки -> Excel» аналогичных пользовательских параметров расчёта на этапе 1 нет.

# 15. Модуль WB -> Скидки -> Excel

## 15.1. Назначение

WB-модуль предназначен для расчёта итоговой скидки и записи её в колонку:

- `Новая скидка`

## 15.2. Входной состав

Один WB run принимает:

- ровно 1 файл цен;
- от 1 до 20 файлов акций.

Поддерживаемый формат:

- только `.xlsx`.

Ограничения размера:

- максимум 25 МБ на один файл;
- максимум 100 МБ суммарно на один WB run;
- максимум 20 promo files.

Если в книге один лист - используется он. Если листов несколько - используется первый лист.

## 15.3. Обязательные колонки

Обязательные колонки файла цен:

- `Артикул WB`;
- `Текущая цена`;
- `Новая скидка`.

Обязательные колонки файлов акций:

- `Артикул WB`;
- `Плановая цена для акции`;
- `Загружаемая скидка для участия в акции`.

## 15.4. Нормализация

Для `Артикул WB`:

- привести к строке;
- trim;
- удалить суффикс `.0`;
- пустое значение считать отсутствующим.

Для числовых значений:

- удалить обычные пробелы;
- удалить неразрывные пробелы;
- заменить запятую на точку;
- нечисловое значение считать отсутствующим.

Одинаковая нормализация обязана применяться и в check, и в process.

## 15.5. Агрегация по акциям

Единица агрегации:

- нормализованный `Артикул WB`.

По каждому артикулу вычисляются:

- `min_discount = MIN(Загружаемая скидка)`;
- `max_plan_price = MAX(Плановая цена для акции)`.

В агрегации участвуют только:

- валидные строки;
- валидные promo files.

## 15.6. Формула расчёта

Обязательная формула:

`calculated_discount = ceil((1 - max_plan_price / current_price) * 100)`

Требования:

- запрещено использовать float;
- обязательно использовать точную десятичную арифметику;
- округление только `ceil`.

## 15.7. Гибридная логика WB

Сначала вычисляется:

`final_discount_pre_threshold = MIN(min_discount, calculated_discount)`

Далее применяется обязательный порядок правил:

1. Если артикул отсутствует во всех валидных файлах акций:
   - `final_discount = fallback_no_promo_percent`.
2. Иначе если `final_discount_pre_threshold > threshold_percent`:
   - `final_discount = fallback_over_threshold_percent`.
3. Иначе:
   - `final_discount = final_discount_pre_threshold`.

Порядок менять нельзя.

## 15.8. Ограничения результата

Итоговая скидка:

- всегда целое число;
- допустимый диапазон - от 0 до 100 включительно;
- выход за диапазон должен отражаться в warning/error semantics по строке.

## 15.9. Управляемые параметры WB

На этапе 1 управляемыми параметрами WB являются только:

- `wb_threshold_percent` - максимальный порог скидки;
- `wb_fallback_no_promo_percent` - скидка на товары, отсутствующие в акциях;
- `wb_fallback_over_threshold_percent` - скидка на товары, у которых акционная скидка превышает порог.

Иных управляемых параметров WB на этапе 1 нет.

Максимальный порог скидки:

- используется как порог сравнения с акционной скидкой товара;
- не является просто справочным значением;
- не трактуется автоматически как общий абсолютный потолок для любой скидки.

Скидка на товары, у которых акционная скидка превышает порог, является целевым значением скидки для ветки «превышен порог».

Скидка на товары, отсутствующие в акциях, является целевым значением скидки для ветки «товар отсутствует в акциях».

## 15.10. Каскад применения параметров WB

Для каждого параметра:

- сначала используется локальное значение магазина / кабинета;
- если оно не задано - системное значение по умолчанию.

Для каждой операции система должна сохранять:

- фактически применённое значение;
- источник значения: системное или магазинное.

## 15.11. Check WB

На этапе check WB система обязана:

- проверить состав файлов;
- открыть и провалидировать файл цен и файлы акций;
- выполнить нормализацию;
- выявить невалидные строки;
- агрегировать валидные акции;
- полностью промоделировать результат по каждой строке;
- сформировать summary audit;
- сформировать detail audit;
- сформировать краткий итог;
- не создавать output file;
- ничего не записывать в workbook.

## 15.12. Process WB

На этапе process WB система обязана:

- выполнить обработку только по допустимой проверке-основанию с учётом правил актуальности проверки;
- выполнить полный расчёт;
- изменить только колонку `Новая скидка`;
- сохранить output workbook;
- сформировать summary audit;
- сформировать detail audit;
- сформировать краткий итог.

## 15.13. Что запрещено менять в WB workbook

Запрещено:

- менять любые колонки, кроме `Новая скидка`;
- менять порядок строк;
- разрушать формулы;
- разрушать пригодность workbook к дальнейшему использованию;
- использовать float в финальном расчёте.

## 15.14. Критические ошибки WB

К критическим ошибкам относятся как минимум:

- не загружен файл цен;
- не загружен ни один файл акций;
- файл не открывается как `.xlsx`;
- книга повреждена;
- нет обязательной колонки файла цен;
- все файлы акций невалидны;
- дубли в файле цен по `Артикул WB`;
- невозможность безопасной записи в `Новая скидка`.

# 16. Модуль Ozon -> Скидки -> Excel

## 16.1. Назначение

Ozon-модуль обрабатывает один Excel-файл акции формата:

- `Эластичный бустинг`.

На этапе 1 система не рассчитывает «скидку в процентах» как отдельный результат.

По каждой строке система должна определить:

- участвует ли товар в акции;
- если участвует - какую итоговую цену акции нужно записать.

## 16.2. Входной состав

Один Ozon run принимает:

- ровно 1 файл `.xlsx`.

Книга должна содержать лист:

- `Товары и цены`.

Если такого листа нет - это критическая ошибка.

Строки данных читаются начиная со строки 4. Строки 1-3 считаются служебной частью шаблона.

## 16.3. Обязательные колонки

В логике Ozon обязательны колонки:

- J;
- K;
- L;
- O;
- P;
- R.

Смысл колонок:

- J - минимально допустимая цена;
- O - минимальная буст-цена;
- P - максимальная буст-цена;
- R - остаток;
- K - признак участия в акции;
- L - итоговая цена акции.

## 16.4. Нормализация

Для J, O, P, R:

- привести данные к нормализованному числовому виду;
- нечисловое значение считать отсутствующим;
- логика решения должна использовать только нормализованные значения.

## 16.5. Разрешённые изменения

Разрешено изменять только:

- K;
- L.

Запрещено изменять:

- J;
- O;
- P;
- R;
- любые другие колонки;
- структуру шаблона;
- листы книги;
- служебные области.

## 16.6. Каноническая логика решения по строке Ozon

Обязательный порядок правил:

Правило 1 - `missing_min_price`.

Если J отсутствует:

- K = пусто;
- L = пусто;
- reason = `missing_min_price`.

Правило 2 - `no_stock`.

Если R отсутствует или <= 0:

- K = пусто;
- L = пусто;
- reason = `no_stock`.

Правило 3 - `no_boost_prices`.

Если O и P одновременно отсутствуют:

- K = пусто;
- L = пусто;
- reason = `no_boost_prices`.

Правило 4 - `use_max_boost_price`.

Если P присутствует и P >= J:

- K = `Да`;
- L = P;
- reason = `use_max_boost_price`.

Правило 5 - `use_min_price`.

Если:

- P присутствует;
- O присутствует;
- P < J;
- O >= J;

то:

- K = `Да`;
- L = J;
- reason = `use_min_price`.

Правило 6 - `below_min_price_threshold`.

Если O присутствует и O < J:

- K = пусто;
- L = пусто;
- reason = `below_min_price_threshold`.

Правило 7 - `insufficient_ozon_input_data`.

Иначе:

- K = пусто;
- L = пусто;
- reason = `insufficient_ozon_input_data`.

Порядок правил менять нельзя.

## 16.7. Что записывается в ячейки Ozon

Колонка K:

- если товар участвует в акции, в K записывается строго `Да`;
- если товар не участвует, в K записывается пустая ячейка.

Колонка L:

- если есть итоговая цена акции, в L записывается числовое значение;
- если товар не участвует, в L записывается пустая ячейка.

Запрещено:

- писать в K что-либо, кроме `Да` или пустого значения;
- писать в L текстовые суррогаты чисел;
- записывать `0` вместо пустого значения там, где должна быть пустая ячейка.

## 16.8. Check Ozon

На этапе check Ozon система обязана:

- проверить наличие ровно одного файла;
- проверить наличие листа `Товары и цены`;
- проверить обязательные колонки по буквам;
- проверить пригодность workbook к безопасной обработке;
- прочитать строки с 4-й строки;
- нормализовать J, O, P, R;
- определить решение по каждой строке;
- сформировать summary audit;
- сформировать detail audit;
- сформировать краткий итог;
- не создавать output file;
- ничего не записывать в workbook.

## 16.9. Process Ozon

На этапе process Ozon система обязана:

- выполнить обработку только по допустимой проверке-основанию с учётом правил актуальности проверки;
- открыть workbook;
- проверить лист и колонки;
- проверить возможность безопасного сохранения output workbook;
- выполнить логику решения по каждой строке;
- записать только K и L;
- сохранить output workbook;
- сформировать summary audit;
- сформировать detail audit;
- сформировать краткий итог.

## 16.10. Критические ошибки Ozon

К критическим ошибкам относятся как минимум:

- файл не загружен;
- загружено более одного файла;
- книга повреждена;
- отсутствует лист `Товары и цены`;
- отсутствуют обязательные колонки по буквам;
- невозможность сохранить выходной файл.

## 16.11. Ключевое отличие Ozon от WB

В модуле Ozon на этапе 1 отсутствуют пользовательски изменяемые параметры расчёта скидок.

Следовательно:

- в Ozon нет пользовательского каталога параметров скидки, аналогичного WB;
- параметры расчёта WB не должны переноситься на Ozon;
- экранная модель Ozon симметрична WB, но параметрическая модель - нет.

# 17. Объяснимость результата и карточка операции

## 17.1. Краткий итог после проверки

После проверки система должна показывать:

- общий статус проверки;
- количество ошибок;
- количество предупреждений;
- краткое описание основных проблем;
- переход к подробной детализации.

## 17.2. Краткий итог после обработки

После успешной обработки на рабочем экране показывается:

- статус операции;
- краткий итог;
- количество обработанных строк;
- наличие предупреждений;
- прямая ссылка на итоговый файл;
- переход в карточку операции.

Если пользователю нужно подробно понять, что произошло во время расчёта, он переходит в карточку операции.

## 17.3. Карточка операции

На этапе 1 должна быть реализована полноценная рабочая карточка операции.

Карточка операции должна быть главным экраном разбора результата для пользователя, но без перегруза внутренними техническими деталями.

Карточка операции должна содержать:

- видимый номер операции;
- маркетплейс;
- модуль;
- режим: Excel / будущий API;
- магазин / кабинет;
- тип операции: проверка / обработка;
- статус;
- кто запустил;
- дата и время запуска / завершения;
- входные файлы и их версии;
- выходной файл, если есть;
- проверка-основание для обработки;
- применённые параметры;
- версия бизнес-логики;
- краткий итог;
- ошибки и предупреждения;
- детализация по строкам;
- подтверждение предупреждений: кто, когда, что подтвердил;
- ссылка на аудит и техжурнал при наличии проблем.

## 17.4. Детализация по строкам

В карточке операции должна быть пользовательская детализация по строкам.

Детализация должна показывать:

- номер строки;
- артикул / идентификатор товара;
- статус строки;
- причина решения;
- ошибка / предупреждение;
- проблемное поле;
- итоговое значение, если применимо;
- reason/result code;
- человекочитаемое пояснение.

Полная техническая трассировка по каждой строке не отображается в карточке операции по умолчанию и относится к техническому журналу или расширенной диагностике для пользователей с отдельным правом.

## 17.5. Detail audit и экспорт детализации

Подробная детализация должна быть доступна:

- в интерфейсе;
- в экспортируемом отчёте Excel.

Система должна хранить и отображать по каждой обработанной строке итоговую причину принятого решения в виде:

- нормализованного reason/result code;
- человекочитаемого пояснения.

Требование распространяется как минимум на:

- detail audit;
- карточку операции;
- экспорт детализации.

# 19. UX-модель этапа 1

Интерфейс этапа 1 проектируется по модели desktop-first.

Основная работа с Excel-файлами, таблицами, операциями, товарами, магазинами, ролями и настройками выполняется с компьютера.

Интерфейс должен быть читаемым на планшете и телефоне для просмотра статусов, операций, ошибок и кратких результатов.

Полноценная мобильная работа с файлами, администрированием и настройкой прав не является обязательной для этапа 1.

Сложные сценарии оформляются отдельными экранами / страницами.

Простые подтверждения, предупреждения и короткие действия могут выполняться через модальные окна.

Полная mobile-first адаптация не входит в обязательный объём этапа 1.

# 20. Аудит, технический журнал и история

## 20.1. Разделение контуров

Операции, аудит и технический журнал - три разных контура системы, не заменяющие друг друга.

- Операции фиксируют бизнес-действия и их результаты.
- Аудит фиксирует значимые действия пользователей и администраторов.
- Технический журнал фиксирует системные события, ошибки и сбои.

## 20.2. Аудит действий

На этапе 1 аудит фиксирует значимые действия пользователей и администраторов:

- запуск проверки;
- запуск обработки;
- подтверждение предупреждений;
- изменение параметров WB;
- создание / изменение магазина;
- изменение пользователей;
- изменение ролей;
- изменение прав;
- изменение доступов к магазинам;
- изменение API-блока;
- скачивание итогового файла, если требуется контролировать доступ к файлам;
- иные критичные действия.

Аудиторские записи:

- формируются автоматически;
- хранятся как неизменяемая история;
- не подлежат редактированию;
- не подлежат обычному удалению через пользовательские и административные интерфейсы.

## 20.3. Технический журнал

Технический журнал отдельно фиксирует системные ошибки:

- ошибки чтения Excel;
- ошибки сохранения файлов;
- сбои обработки;
- системные исключения;
- ошибки базы данных;
- ошибки будущих API-подключений.

Записи технического журнала:

- не редактируются;
- не удаляются обычными средствами через пользовательские и административные интерфейсы;
- могут очищаться только по регламенту хранения.

## 20.4. Карточки записей

И аудит, и техжурнал обязаны иметь:

- список записей;
- карточку записи;
- фильтрацию;
- поиск;
- переходы к связанным сущностям, если применимо.

# 21. Удаление, блокировка и архивирование

На этапе 1 используется смешанная модель удаления.

Если сущность уже участвовала в работе системы, она не удаляется физически, а блокируется, деактивируется или архивируется.

Если сущность создана ошибочно и ещё нигде не использовалась, её можно физически удалить.

Операции, аудит и технический журнал обычным способом не удаляются.

Файлы могут удаляться по сроку хранения, но запись об операции и связь с файлом остаются.

Пользователи, участвовавшие в системе, подлежат блокировке или архивированию. Обычное физическое удаление таких пользователей не допускается.

Магазины / кабинеты, участвовавшие в операциях или настройках, подлежат архивированию или деактивации. Физическое удаление таких сущностей не допускается.

Роли, уже использованные в назначениях или истории, не подлежат обычному физическому удалению.

# 22. Execution layer, сбои, восстановление, обновления и резервное копирование

## 22.1. Execution layer

На этапе 1 допускается синхронное выполнение части операций.

При этом должен существовать единый execution layer, совместимый с будущим переводом операций в:

- фоновые задачи;
- очередь;
- async execution.

Это обязательно для будущих:

- API-процессов;
- тяжёлых расчётов;
- автоматических сценариев;
- производства;
- поставок;
- отчётов.

## 22.2. Сбои и прерванные операции

Операция, прерванная из-за сбоя приложения, сервера, базы данных или файлового хранилища, должна переводиться в явный статус прерванной / сбойной операции.

На этапе 1 автоматическое продолжение выполнения прерванной операции после восстановления системы не допускается.

Архитектура execution layer и модели операций должна допускать возможность последующего внедрения механизма безопасного возобновления для отдельных типов операций в будущих этапах.

## 22.3. Восстановление

Система и исполнительная документация должны предусматривать обязательный базовый механизм восстановления после сбоя:

- приложения;
- сервера;
- базы данных;
- файлового хранилища.

Механизм восстановления должен включать:

- порядок перезапуска;
- проверку целостности;
- проверку доступности веб-панели;
- правила обработки незавершённых операций.

## 22.4. Обновление production

На этапе 1 обновление рабочей системы может выполняться напрямую в production-контуре.

Исполнительная документация обязана содержать контролируемый регламент обновления:

- резервное копирование перед обновлением;
- порядок применения обновления;
- проверку работоспособности после обновления;
- действия по откату при неуспехе.

## 22.5. Резервное копирование

На этапе 1 резервное копирование является обязательной частью эксплуатационного контура системы.

Обязательному резервному копированию подлежат как минимум:

- PostgreSQL;
- серверное файловое хранилище.

Исполнительная документация должна содержать порядок восстановления:

- после сбоя;
- после неуспешного обновления;
- после утраты части данных.

## 22.6. Системные уведомления

На этапе 1 система должна иметь минимальный контур системных уведомлений о критичных проблемах и состояниях, требующих внимания.

Уведомления должны формироваться как минимум для:

- сбоев операций;
- ошибок сохранения файлов;
- критичных ошибок подключений;
- иных проблем, влияющих на работоспособность системы или корректность пользовательских сценариев.

# 24. Приёмка этапа 1 и контрольные файлы

## 24.1. Общий принцип приёмки

- в спецификации ролей и прав;
- в спецификации операций;
- в файловом контуре;
- в эксплуатационной документации;
- в тестовой документации;
- в приёмочной документации;
- в агентных инструкциях и шаблонах задач.

## 25.3. Недопустимые ссылки

Исполнительная документация не должна ссылаться на:

- «старую программу» как на источник правил без описания этих правил внутри документации;
- переписку;
- устные договорённости;
- «по аналогии»;
- скрытый контекст;
- неутверждённые предположения.

# 26. Требования к реализации и проектированию агентами Codex CLI

## 26.1. Общий принцип

Исполнительная проектная документация должна быть подготовлена не только для человека-исполнителя, но и для реализации проекта агентами Codex CLI.

Документация должна быть полностью адаптирована под агентную схему разработки, в которой есть:

- оркестратор Codex CLI;
- проектировщик исполнительной проектной документации Codex CLI;
- внутренний проектировщик Codex CLI;
- разработчик Codex CLI;
- аудитор Codex CLI;
- тестировщик Codex CLI;
- техрайтер Codex CLI.

## 26.2. Разделение проектировщиков

Проектировщик исполнительной проектной документации Codex CLI и внутренний проектировщик Codex CLI - разные роли.

Проектировщик исполнительной проектной документации Codex CLI работает до начала реализации и обязан подготовить самодостаточный комплект проектной документации по настоящему ТЗ.

Внутренний проектировщик Codex CLI работает внутри процесса реализации и подключается оркестратором только при необходимости:

- если обнаружен пробел в документации;
- если нужна детализация параметров системы;
- если нужно уточнить проектное решение без изменения утверждённой бизнес-логики;
- если нужно подготовить проектное решение для будущего модуля или технического участка.

Внутренний проектировщик Codex CLI не имеет права менять утверждённые бизнес-правила, модель прав, модель операций, файловый контур, reason/result codes, сценарии WB/Ozon или границы этапа без явного согласования.

## 26.3. Оркестратор Codex CLI

Оркестратор Codex CLI является главным управляющим агентом процесса разработки.

Оркестратор должен:

- принимать исходное ТЗ и исполнительную документацию;
- разбивать работу на этапы, задачи и подзадачи;
- назначать задачи профильным агентам;
- получать результаты от агентов;
- проверять полноту результата;
- передавать результат на аудит, тестирование и документирование;
- фиксировать нерешённые вопросы;
- не допускать реализации на основе домысливания;
- при пробеле в документации передавать задачу внутреннему проектировщику Codex CLI;
- поддерживать непрерывный рабочий цикл до ручной остановки заказчиком.

Работа оркестратора должна быть организована как постоянно активная управляющая сессия, находящаяся в ожидании результатов агентов и новых задач, если заказчик явно не остановил процесс.

## 26.4. Параллельная работа

При технической необходимости и при отсутствии конфликтов работу можно вести в несколько параллельных потоков.

Параллельные потоки допустимы, например, для:

- backend;
- frontend;
- модель данных и миграции;
- бизнес-логика WB/Ozon;
- тесты;
- эксплуатационная документация;
- аудит безопасности и соответствия ТЗ.

Параллельная работа запрещена, если задачи одновременно меняют конфликтующие части:

- одну и ту же миграцию;
- одну и ту же модель данных;
- одну и ту же бизнес-логику;
- один и тот же экран;
- один и тот же контур прав;
- один и тот же файловый или операционный контур.

## 26.5. Обязательный состав агентной проектной документации

Исполнительная проектная документация должна содержать отдельный комплект документов для агентной реализации:

1. `AGENTS.md` - основные инструкции для Codex CLI в репозитории.
2. `ORCHESTRATION.md` - порядок работы оркестратора.
3. `AGENT_ROLES_MATRIX.md` - матрица ролей агентов.
4. `TASK_TEMPLATES.md` - шаблоны задач агентам.
5. `HANDOFF_TEMPLATES.md` - шаблоны передачи результата между агентами.
6. `ACCEPTANCE_CHECKLISTS.md` - чек-листы приёмки задач.
7. `GAP_REGISTER.md` - реестр пробелов, вопросов и противоречий.
8. `ADR_LOG.md` - журнал архитектурных решений.
9. `PARALLEL_WORK_RULES.md` - правила параллельной работы.
10. `AUDIT_PROTOCOL.md` - правила аудита кода, архитектуры, миграций, безопасности и соответствия ТЗ.
11. `TEST_PROTOCOL.md` - правила тестирования.
12. `DOCUMENTATION_UPDATE_PROTOCOL.md` - правила обновления документации.
13. `RELEASE_AND_UPDATE_RUNBOOK.md` - порядок релизов, обновлений и отката.
14. `PROJECT_STRUCTURE.md` - структура проекта и правила размещения файлов.
15. `MODULE_SPECIFICATIONS/` - спецификации модулей, сценариев, экранов и сущностей.

Документация должна быть написана так, чтобы агент мог выполнять задачу без обращения к переписке, устным договорённостям или скрытому контексту.

## 26.6. Входные и выходные данные агентов

Для каждого агента документация должна явно описывать:

- назначение агента;
- границы ответственности;
- какие документы он получает на вход;
- какие файлы / разделы проекта он может изменять;
- какие файлы / разделы ему запрещено изменять;
- какой результат он обязан вернуть;
- в каком формате возвращается результат;
- какие проверки он обязан выполнить;
- кому передаётся результат после завершения;
- что делать при обнаружении пробела или противоречия.

Минимальная матрица агентов:

- оркестратор Codex CLI: вход - ТЗ, проектная документация, состояние репозитория; выход - план задач, маршрутизация, контроль, итоговая сводка;
- проектировщик исполнительной проектной документации Codex CLI: вход - итоговое ТЗ; выход - полный комплект исполнительной проектной документации;
- внутренний проектировщик Codex CLI: вход - конкретный пробел / задача; выход - проектное уточнение, ADR или вопрос на согласование;
- разработчик Codex CLI: вход - задача, спецификация, чек-лист; выход - код, миграции, тесты, отчёт о выполнении;
- аудитор Codex CLI: вход - код, документация, чек-листы; выход - аудиторское заключение, замечания, решение pass/fail;
- тестировщик Codex CLI: вход - спецификация, тест-протокол, контрольные данные; выход - тесты, результаты прогонов, список дефектов;
- техрайтер Codex CLI: вход - изменения, решения, отчёты; выход - обновлённая документация, changelog, фиксация этапов.

## 26.7. Шаблоны агентной работы

### 26.7.1. Шаблон задачи агенту

Каждая задача агенту должна содержать:

- идентификатор задачи;
- тип задачи;
- цель;
- входные документы;
- связанные требования ТЗ;
- разрешённые файлы / области изменения;
- запрещённые файлы / области изменения;
- ожидаемый результат;
- критерии завершённости;
- обязательные проверки;
- формат отчёта;
- получатель результата.

### 26.7.2. Шаблон результата агента

Результат агента должен содержать:

- что сделано;
- какие файлы изменены;
- какие требования закрыты;
- какие тесты / проверки выполнены;
- какие проблемы обнаружены;
- какие вопросы требуют решения;
- что передать следующему агенту.

### 26.7.3. Шаблон передачи между агентами

Передача результата между агентами должна содержать:

- исходная задача;
- выполненный объём;
- ссылки на изменённые файлы;
- текущий статус;
- ограничения;
- замечания;
- следующий ожидаемый шаг.

### 26.7.4. Шаблон записи в GAP_REGISTER

Запись в реестре пробелов должна содержать:

- идентификатор пробела;
- где обнаружен пробел;
- какое требование затронуто;
- почему нельзя продолжать без решения;
- возможные варианты решения, если они есть;
- рекомендуемый вариант, если агент может предложить его без изменения бизнес-логики;
- кто должен принять решение;
- статус решения.

### 26.7.5. Шаблон аудиторского заключения

Аудиторское заключение должно содержать:

- проверенную область;
- соответствие ТЗ;
- соответствие проектной документации;
- найденные нарушения;
- риски;
- обязательные исправления;
- рекомендации;
- итог: pass / fail / pass with remarks.

## 26.8. Запрет на домысливание агентами

Ни один агент Codex CLI не имеет права самостоятельно принимать бизнес-решения, если они не зафиксированы в ТЗ, дополнениях или исполнительной проектной документации.

Запрещено без согласования менять:

- бизнес-логику WB;
- бизнес-логику Ozon;
- модель прав;
- модель магазинов / кабинетов;
- модель операций;
- файловый контур;
- reason/result codes;
- правила проверки / обработки;
- границы этапа 1;
- статус Excel как штатного режима.

При обнаружении неоднозначности агент должен:

1. остановить реализацию спорного участка;
2. зафиксировать вопрос в `GAP_REGISTER.md`;
3. передать вопрос оркестратору;
4. оркестратор передаёт вопрос внутреннему проектировщику Codex CLI или заказчику;
5. работа продолжается только после фиксации решения.

## 26.9. Требования к AGENTS.md

В корне репозитория должен быть файл `AGENTS.md`.

`AGENTS.md` должен содержать:

- краткое назначение проекта;
- ссылку на итоговое ТЗ и исполнительную документацию;
- запрет на домысливание;
- правила изменения кода;
- правила изменения документации;
- правила работы с миграциями;
- правила тестирования;
- правила аудита;
- правила фиксации пробелов;
- требования к отчёту агента;
- запрет на изменение бизнес-логики без утверждённого основания;
- правила параллельной работы.

При наличии поддиректорий с особыми правилами допускаются дополнительные вложенные `AGENTS.md`, уточняющие инструкции для конкретного модуля или области.

`AGENTS.md` не должен противоречить настоящему ТЗ и исполнительной проектной документации.

## 26.10. Документация как обязательный результат разработки

Каждая значимая задача разработки считается незавершённой, если не обновлена связанная документация.

Техрайтер Codex CLI должен фиксировать:

- что было реализовано;
- какие требования закрыты;
- какие документы изменены;
- какие решения приняты;
- какие пробелы закрыты;
- какие ограничения остались;
- какие шаги следующие.

По итогам каждого этапа должен быть сформирован отчёт:

- выполненные задачи;
- закрытые требования;
- незакрытые требования;
- обнаруженные отклонения;
- тесты и результаты;
- готовность к приёмке.

# 27. Минимальные нефункциональные требования этапа 1

## 27.1. Безопасность

Система должна обеспечивать:

- аутентификацию пользователей;
- разграничение доступа по ролям, правам, доступам и магазинам / кабинетам;
- защиту API-ключей и чувствительных параметров;
- ограничение видимости технических деталей;
- неизменяемость аудита и завершённых операций;
- невозможность скачивания файлов пользователями без соответствующих прав и объектного доступа.

## 27.2. Надёжность

Система должна обеспечивать:

- корректное завершение операций;
- явную фиксацию прерванных / сбойных операций;
- отсутствие автоматического продолжения сбойных операций на этапе 1;
- сохранение метаданных операций после истечения срока хранения файлов;
- резервное копирование базы данных и файлового хранилища;
- документированный порядок восстановления.

## 27.3. Расширяемость

Система должна быть спроектирована так, чтобы в будущем добавить:

- скидки по API;
- цены по Excel и API;
- остатки;
- продажи;
- расчёт среднесуточных продаж;
- расчёт потребности;
- производство;
- поставки;
- складские состояния;
- журналы операторов;
- закупки;
- автоматические сценарии;
- фоновые задачи и очереди.

Добавление будущих модулей не должно требовать перепроектирования базовых сущностей этапа 1: пользователей, ролей, прав, магазинов, товаров, операций, файлов, аудита и техжурнала.

# 28. Заключительная фиксация

Настоящее итоговое ТЗ является единой обязательной основой для проектирования, реализации, аудита, тестирования и приёмки этапа 1.

Все положения, относящиеся к обязательному объёму этапа 1, должны быть реализованы либо явно отражены в исполнительной проектной документации как обязательные проектные решения.

Все положения, относящиеся к будущим этапам, должны быть заложены архитектурно в той мере, в которой это влияет на базовую модель платформы.

Особое внимание при проектировании должно быть уделено:

- корректной логике «Проверить / Обработать»;
- неизменяемости завершённых операций;
- связям операций с версиями файлов;
- параметрам WB на уровне магазина / кабинета;
- отсутствию пользовательских параметров Ozon на этапе 1;
- объектным ограничениям по магазинам / кабинетам;
- роли владельца;
- разделению операций, аудита и технического журнала;
- формальной приёмке по контрольным файлам;
- адаптации исполнительной документации под агентную реализацию через Codex CLI.
