Мой вывод: **Этап 2.2 нужно проектировать не как новый расчёт, а как Ozon API-адаптер к уже утверждённой Ozon Excel-логике по акции “Эластичный бустинг”**. Основной риск — не формула, а корректное сопоставление полей API Ozon с колонками Excel и безопасная загрузка результата в акцию.

## 1. Что уже есть по Ozon Excel

В проекте уже описана и реализована бизнес-логика Ozon Excel для одного формата акции: **`Эластичный бустинг`**. Вход — один `.xlsx`, лист `Товары и цены`, данные с 4-й строки. Процесс имеет право менять только колонки **K** и **L**: K — участие, L — итоговая цена в акции. Управляемых пользователем параметров скидки для Ozon Excel нет. ([GitHub][1])

Каноническая логика расчёта состоит из 7 правил:

|  № | Условие                                 | Результат         |
| -: | --------------------------------------- | ----------------- |
|  1 | Нет минимально допустимой цены `J`      | K/L пусто         |
|  2 | Нет остатка или остаток `R <= 0`        | K/L пусто         |
|  3 | Нет обеих буст-цен `O` и `P`            | K/L пусто         |
|  4 | `P >= J`                                | K = `Да`, L = `P` |
|  5 | `P < J`, при этом `O >= J`              | K = `Да`, L = `J` |
|  6 | `O < J`                                 | K/L пусто         |
|  7 | Остальные неполные/неоднозначные данные | K/L пусто         |

Это и надо брать как неизменяемое ядро Stage 2.2. API-часть должна только собрать эквивалентные входные данные, прогнать тот же расчёт и загрузить рассчитанную цену участия.

## 2. Что важно по API Ozon

Для промоакций Ozon есть отдельный контур `/v1/actions/*`: список доступных акций, список товаров-кандидатов, список участвующих товаров, добавление и удаление товаров из акции. В публичной документации/спецификациях фигурируют методы `/v1/actions`, `/v1/actions/candidates`, `/v1/actions/products`, `/v1/actions/products/activate`, `/v1/actions/products/deactivate`. ([docs.ozon.ru][2])

Для выбранной акции товары-кандидаты можно получать через `/v1/actions/candidates`; запрос содержит `action_id`, `limit`, `offset`, а ответ включает товарные поля вроде `id`, `price`, `action_price`, `max_action_price`, `add_mode`, `min_stock`, `stock`. Для уже участвующих товаров используется `/v1/actions/products`. ([postman.com][3])

Ключевая свежая деталь: Ozon добавил в ответы `/v1/actions/candidates` и `/v1/actions/products` поля, связанные именно с эластичным бустингом: `current_boost`, `price_min_elastic`, `price_max_elastic`, `min_boost`, `max_boost`. Это сильно упрощает Stage 2.2, потому что API теперь потенциально отдаёт аналоги минимальной и максимальной буст-цены, которые в Excel находятся в колонках **O** и **P**. ([Telegram][4])

Отдельно в Ozon API появился параметр `prices.manage_elastic_boosting_through_price` в `/v1/product/import/prices`. Я бы **не включал его в Stage 2.2 v1**, потому что это другой механизм управления через цену товара, а не прямой перенос Excel-логики “рассчитать цену участия и загрузить в акцию”. ([Telegram][5])

## 3. Предлагаемый scope Stage 2.2

### В scope

**Только Ozon API → акция “Эластичный бустинг”.**

Функционально:

1. Подключение Ozon API для магазина/кабинета.
2. Получение списка акций Ozon.
3. Фильтрация/выбор только акции типа **Эластичный бустинг**.
4. Выгрузка товаров-кандидатов и уже участвующих товаров по выбранной акции.
5. Расчёт по тем же 7 правилам, что Ozon Excel.
6. Формирование отчёта/результата расчёта.
7. Подтверждённая загрузка рассчитанных цен участия в Ozon через API.
8. Логирование операций, аудит, techlog, безопасные snapshots без секретов.

### Не в scope

1. Обычные акции Ozon, Hot Sale, акции продавца, seller-actions.
2. Автоматическое удаление товаров из акции.
3. Изменение базовых цен товара.
4. Использование `/v1/product/import/prices` для управления эластичным бустингом.
5. Изменение Stage 1 Ozon Excel.
6. Изменение WB API Stage 2.1.
7. Пользовательские параметры скидки для Ozon — в Excel-логике их нет, значит в Stage 2.2 их тоже не вводить.

## 4. Главная архитектурная рекомендация

Сделать новый модуль:

```text
apps/discounts/ozon_api/
```

А бизнес-логику Ozon Excel вынести или переиспользовать как shared-ядро:

```text
apps/discounts/ozon_shared/
```

Идея такая:

```text
Ozon API rows
    ↓
adapter: API fields → canonical Ozon discount input
    ↓
same Ozon Excel decision engine
    ↓
calculation result snapshot
    ↓
user confirmation
    ↓
Ozon API upload
```

То есть Stage 2.2 не должен дублировать расчёт внутри API-модуля. Он должен использовать тот же `decide_row`/эквивалентную функцию, которая сейчас обслуживает Excel.

## 5. Предлагаемое сопоставление полей

Предварительная карта такая:

| Excel-поле | Смысл в текущей логике     | Предлагаемый API-источник                                                                                                            | Статус              |
| ---------- | -------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ | ------------------- |
| `J`        | минимально допустимая цена | нужно подтвердить по реальному API-ответу; кандидат — отдельное поле минимальной цены товара или утверждённый источник из price/info | **GAP**             |
| `O`        | минимальная буст-цена      | `price_min_elastic`                                                                                                                  | выглядит подходящим |
| `P`        | максимальная буст-цена     | `price_max_elastic`                                                                                                                  | выглядит подходящим |
| `R`        | остаток                    | `stock`                                                                                                                              | выглядит подходящим |
| `K`        | участие                    | генерируется расчётом                                                                                                                | output              |
| `L`        | итоговая цена акции        | генерируется расчётом; затем `action_price` для загрузки                                                                             | output              |

Самый важный GAP — колонка **J**. Нельзя без проверки подставлять туда `max_action_price`: по названию это скорее “максимальная цена для акции”, а в Excel-логике J используется как нижний порог допустимой цены продавца. Нужно взять один реальный sanitized API response по эластичному бустингу и подтвердить, какое поле Ozon соответствует J.

По `O` и `P` ситуация лучше: новые поля `price_min_elastic` и `price_max_elastic` выглядят прямыми кандидатами на минимальную и максимальную буст-цену. Но это тоже нужно закрепить тестовым fixture на реальных данных Ozon.

## 6. Предлагаемый workflow Stage 2.2

### Шаг 1. Ozon API connection

Создать connection block для Ozon API.

Требования:

* хранить `Client-Id` и `Api-Key` только через `protected_secret_ref`;
* не писать ключи в metadata, audit, techlog, snapshots, ошибки;
* статусы подключения: `not_configured`, `configured`, `active`, `check_failed`, `disabled`, `archived`;
* операции API разрешать только для `active`;
* проверка подключения — только read-only endpoint;
* права доступа отдельно от WB.

Это соответствует уже принятому подходу проекта для API-подключений: секреты не попадают в metadata, статусы connection block управляют доступностью операций, API snapshots должны быть безопасными. ([GitHub][6])

### Шаг 2. Получение акций Ozon

Операция:

```text
ozon_api_actions_download
```

Задача:

* получить список акций через Ozon Actions API;
* оставить только акции типа/названия **Эластичный бустинг**;
* сохранить безопасный snapshot;
* дать пользователю выбрать конкретный `action_id`.

Здесь важно не смешивать `/v1/actions/*` и `/v1/seller-actions/*`. Stage 2.2 работает с акцией Ozon “Эластичный бустинг”, а не с акциями продавца.

### Шаг 3. Получение товаров акции

Операция:

```text
ozon_api_elastic_products_download
```

Задача:

* по выбранному `action_id` получить товары-кандидаты через `/v1/actions/candidates`;
* получить уже участвующие товары через `/v1/actions/products`;
* пройти пагинацию через `limit` / `offset`;
* сохранить snapshot строк;
* нормализовать числовые поля;
* отметить источник строки: `candidate`, `active`, возможно `candidate_and_active`.

Нужно сохранять не только товары-кандидаты, но и участвующие товары. Это необходимо для контроля текущего состояния акции и drift-check перед загрузкой.

### Шаг 4. Расчёт

Операция:

```text
ozon_api_elastic_calculation
```

Расчёт должен использовать тот же порядок правил, что Excel:

```text
missing_min_price
no_stock
no_boost_prices
use_max_boost_price
use_min_price
below_min_price_threshold
insufficient_ozon_input_data
```

Результат строки:

| Поле                 | Назначение                          |
| -------------------- | ----------------------------------- |
| `product_id`         | ID товара Ozon                      |
| `offer_id` / SKU     | для отображения и сверки            |
| `action_id`          | выбранная акция                     |
| `min_allowed_price`  | аналог J                            |
| `price_min_elastic`  | аналог O                            |
| `price_max_elastic`  | аналог P                            |
| `stock`              | аналог R                            |
| `decision_code`      | одно из 7 правил                    |
| `participate`        | true/false                          |
| `action_price`       | рассчитанная цена для Ozon          |
| `upload_ready`       | можно ли отправлять в API           |
| `source_snapshot_id` | immutable-ссылка на исходные данные |

Для строк `use_max_boost_price` и `use_min_price`:

```text
upload_ready = true
action_price = calculated L
```

Для остальных:

```text
upload_ready = false
action_price = null
```

### Шаг 5. Отчёт расчёта

Даже при API-загрузке нужен выходной файл/отчёт для контроля. Предлагаю формировать `.xlsx` с результатом:

```text
Ozon Elastic Boosting Calculation Report
```

Колонки:

* магазин;
* кабинет;
* action_id;
* название акции;
* product_id;
* offer_id;
* название товара, если API отдаёт;
* текущая цена;
* текущая цена акции;
* J / минимально допустимая цена;
* O / минимальная буст-цена;
* P / максимальная буст-цена;
* остаток;
* текущий буст;
* min/max boost;
* решение;
* рассчитанная цена;
* upload_ready;
* причина отказа.

Этот Excel не должен быть источником истины для загрузки. Источник истины — immutable calculation snapshot. Файл нужен для аудита и ручной проверки.

### Шаг 6. Подтверждение загрузки

Операция:

```text
ozon_api_elastic_upload
```

Перед загрузкой обязательно:

1. Проверить, что connection всё ещё `active`.
2. Проверить, что action всё ещё существует и является Elastic Boosting.
3. Повторно получить данные по товарам.
4. Сравнить критичные поля с calculation snapshot:

   * J;
   * O;
   * P;
   * R;
   * action status;
   * product eligibility.
5. Заблокировать upload при drift.
6. Заблокировать upload, если нет ни одной `upload_ready` строки.
7. Запросить явное подтверждение пользователя.

Загружать нужно только строки:

```text
upload_ready = true
```

Через `/v1/actions/products/activate` с payload уровня:

```json
{
  "action_id": 123,
  "products": [
    {
      "product_id": 456,
      "action_price": 789
    }
  ]
}
```

Если актуальная документация Ozon требует дополнительные поля, например `stock`, их нужно брать из snapshot и валидировать отдельно. Но базовая логика расчёта от этого не меняется.

## 7. Важное решение: не удалять товары автоматически

Моя рекомендация: **в Stage 2.2 v1 не делать автоматический deactivate для товаров, которые не прошли расчёт**.

Причина: Excel-логика просто оставляет K/L пустыми. Она не говорит “удалить товар из акции”. Поэтому API-эквивалентом безопаснее считать:

* прошёл расчёт → добавить/обновить участие с рассчитанной ценой;
* не прошёл расчёт → ничего не отправлять.

Автоматическое удаление через `/v1/actions/products/deactivate` должно быть отдельной функцией с отдельным подтверждением, отдельным ТЗ и отдельными тестами. Метод деактивации существует в контуре Ozon actions, но включать его в базовый Stage 2.2 рискованно. ([PyPI][7])

## 8. Предлагаемая разбивка задач

### TASK-018 — Ozon API connection

Цель: добавить безопасное подключение Ozon API.

Acceptance criteria:

* можно создать/проверить/отключить Ozon API connection;
* секреты не попадают в БД metadata, логи, audit, techlog;
* upload/download запрещён без active connection;
* права доступа проверяются отдельно;
* connection Ozon не связан с WB connection.

### TASK-019 — Ozon Elastic Boosting actions download

Цель: получить список акций и выбрать только Elastic Boosting.

Acceptance criteria:

* доступен список акций Ozon;
* акции фильтруются до Elastic Boosting;
* пользователь выбирает конкретный action;
* сохраняется безопасный snapshot;
* не используются seller-actions.

### TASK-020 — Ozon Elastic Boosting products download

Цель: получить товары по выбранной акции.

Acceptance criteria:

* candidates выгружаются постранично;
* participating products выгружаются постранично;
* строки нормализуются;
* сохраняются `product_id`, `offer_id`, цены, остатки, elastic-поля;
* пустой результат обрабатывается без падения;
* API-ответы без нужных elastic-полей блокируют расчёт с понятной ошибкой.

### TASK-021 — Ozon Elastic Boosting calculation

Цель: рассчитать участие и цену по той же логике, что Excel.

Acceptance criteria:

* 7 правил полностью совпадают с Ozon Excel;
* есть golden-тест “API fixture → тот же результат, что Excel fixture”;
* нет пользовательских параметров скидки;
* формируется отчёт `.xlsx`;
* создаются operation detail rows;
* calculation snapshot immutable.

### TASK-022 — Ozon Elastic Boosting upload

Цель: загрузить рассчитанные цены в акцию Ozon.

Acceptance criteria:

* upload возможен только после calculation;
* upload требует confirmation;
* перед upload выполняется drift-check;
* пустой upload запрещён;
* загружаются только `upload_ready` строки;
* rejected rows сохраняются с причинами Ozon;
* partial rejection не теряет успешные/ошибочные строки;
* повторная отправка защищена от случайного дубля;
* deactivate не выполняется.

### TASK-023 — UI + acceptance + release readiness

Цель: собрать пользовательский контур и регрессию.

Acceptance criteria:

* UI: подключение Ozon API;
* UI: выбор Elastic Boosting акции;
* UI: download → calculate → review → confirm upload;
* карточка операции показывает исходные данные, результат, файл отчёта, статус upload;
* permissions покрыты тестами;
* Stage 1 Ozon Excel regression проходит без изменений;
* Stage 2.1 WB regression не затронут;
* full suite проходит.

## 10. Новые коды причин

Расчётные reason codes лучше оставить такими же, как в Excel:

```text
missing_min_price
no_stock
no_boost_prices
use_max_boost_price
use_min_price
below_min_price_threshold
insufficient_ozon_input_data
```

API-level коды добавить отдельно:

```text
ozon_api_action_not_elastic
ozon_api_action_not_found
ozon_api_missing_elastic_fields
ozon_api_missing_min_allowed_price_source
ozon_api_product_not_eligible
ozon_api_upload_blocked_by_drift
ozon_api_upload_rejected
ozon_api_upload_partial_rejected
ozon_api_upload_success
ozon_api_auth_failed
ozon_api_rate_limited
ozon_api_timeout
ozon_api_response_invalid
```

Так расчётная логика останется совместимой с Excel, а API-ошибки не будут смешиваться с бизнес-правилами.

## 11. Главные GAP перед финальным ТЗ

1. **Поле J.** Нужно точно определить источник “минимально допустимой цены” в API. Это главный открытый вопрос.
2. **Идентификация Elastic Boosting.** Нужно подтвердить, есть ли стабильный `action_type`/код акции или придётся фильтровать по названию.
3. **Payload activate.** Нужно проверить актуальную схему `/v1/actions/products/activate` для Elastic Boosting: достаточно ли `product_id + action_price` или нужен ещё `stock`.
4. **Batch size/rate limits.** Нужно зафиксировать лимиты Ozon для actions endpoints.
5. **Поведение по уже участвующим товарам.** Моя рекомендация — не удалять их автоматически; только добавлять/обновлять рассчитанные upload-ready строки.
6. **Real sanitized fixture.** Нужен один обезличенный API response из реального кабинета Ozon по Elastic Boosting, чтобы закрыть mapping J/O/P/R.

## 12. Итоговая рекомендация

Формулировка для ТЗ:

> Stage 2.2 реализует Ozon API-контур для акции “Эластичный бустинг”. Расчёт участия и цены должен полностью соответствовать существующей Ozon Excel-логике Stage 1. API-модуль не вводит новую бизнес-логику, а только получает данные из Ozon API, приводит их к каноническому формату J/O/P/R, выполняет существующий расчёт, формирует отчёт и после явного подтверждения пользователя загружает рассчитанные action prices в выбранную акцию Ozon.

Ключевое проектное решение: **расчёт один, источники разные**. Excel и API должны расходиться только адаптерами ввода/вывода, а не правилами принятия решения.
